## 1. Agent 是什么：从"执行"到"行为"的转变

> 📖 来源：`foundations/agent-systems/what-is-agent.mdx`

### 1.1 核心定义

**Agent 是以目标为导向的闭环决策系统，而非一次性计算。**

```
传统程序 / 独立模型          Agent

执行预定义逻辑                持续感知环境
产生一次性输出                自主做出决策
无法长期适应                  采取行动追求目标
不保持状态                    在动态循环中自我修正
```

### 1.2 核心循环：Observe → Think → Act → Reflect → Loop

```
          ┌──────────────────────────────────┐
          │                                  │
          ▼                                  │
    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
    │ Observe  │ →  │  Think   │ →  │   Act    │ →  │ Reflect  │
    │ 感知环境  │    │ 推理决策  │    │ 执行动作  │    │ 评估结果  │
    └──────────┘    └──────────┘    └──────────┘    └────┬─────┘
          │                                              │
          │              目标未达成 → 回到 Observe         │
          │              目标已达成 → 最终输出             │
          └──────────────────────────────────────────────┘
```

**关键理解**：Agent 的智能不由某一次决策定义，而由**随时间迭代改进行动的能力**定义。Agent 从根本上是一个**持续运行的闭环系统**，不是一次函数调用。

### 1.3 实例：每日新闻观察者

| 阶段 | 动作 |
|------|------|
| **Observe** | 接收请求"总结最新 AI 新闻"，从多个来源抓取文章 |
| **Think** | 判断哪些文章相关，按主题筛选，决定内容组织方式 |
| **Act** | 检索、解析、总结选中文章，生成结构化报告 |
| **Reflect** | 评估结果是否完整连贯，不足则调整后重试 |

### 1.4 Agent ≠ 传统脚本

| 维度 | 传统脚本 | Agent |
|------|---------|-------|
| **决策方式** | 固定步骤序列 | 根据中间结果动态决定每一步 |
| **上下文感知** | 依赖预定义规则 | 解释输入并自适应调整行为 |
| **控制流** | 静态、不可变 | 可循环、修正、分支 |
| **数据处理** | 确定性处理结构化数据 | 用模型处理非结构化数据（总结、筛选、推理） |

### 1.5 现代 Agent 的四层架构

```
┌─────────────────────────────────────────────┐
│          LLM（推理核心 / "大脑"）              │
│          解释输入，生成计划，决定做什么         │
├─────────────────────────────────────────────┤
│          工具（行动层 / "四肢"）               │
│          与外部系统交互，执行具体动作           │
├─────────────────────────────────────────────┤
│          记忆（上下文层）                     │
│          跨步骤连续性，上下文保留和适应         │
├─────────────────────────────────────────────┤
│          框架（编排层 / "神经系统"）            │
│          解析模型输出 → 结构化函数调用 → 执行    │
└─────────────────────────────────────────────┘
```

> 💡 **核心心智模型**：LLM 是大脑，工具是四肢，后端系统是神经系统——把抽象推理转化为具体操作。

### 1.6 Agent 的三大取舍

| 特性 | 带来的能力 | 带来的代价 |
|------|-----------|-----------|
| **循环** | 多步推理、自我修正 | 延迟 ↑、Token ↑、复杂度 ↑ |
| **自主性** | 无需持续人工监督 | 错误决策风险、误解目标风险 |
| **工具依赖** | 扩展能力边界 | API 故障、延迟尖峰直接影响系统稳定性 |

## 2. Agent 系统四要素：环境 · 感知 · 行动 · 自主

> 📖 来源：`foundations/the-agent-system.mdx`

### 2.1 四个判定标准

一个有意义的"Agent 系统"必须同时具备：

```
┌──────────────┐
│  Environment │ ← 系统运行其中的那部分世界
│     环境      │
├──────────────┤
│  Perception  │ ← 系统如何了解这个环境
│     感知      │
├──────────────┤
│    Action    │ ← 系统如何改变或查询这个环境
│     行动      │
├──────────────┤
│   Autonomy   │ ← 系统在感知和行动之间承担多少决策
│     自主      │
└──────────────┘
```

### 2.2 红线：不是所有 LLM 应用都是 Agent 系统

| ✅ 是 Agent 系统 | ❌ 不是 Agent 系统 |
|------------------|-------------------|
| 有明确的任务或目标 | 只有聊天框 |
| 循环运行，非一次响应 | 一问一答 |
| 可访问工具、API、文件等 | 无外部交互 |
| 跨步骤有重要的记忆或状态 | 无状态 |

### 2.3 系统边界四问

设计 Agent 系统时必须明确：

1. **智能体能感知什么？** —— 环境边界
2. **它能做什么？** —— 行动边界
3. **它需要独立负责决定什么？** —— 自主边界
4. **什么时候必须由人介入？** —— 人机边界

## 3. Agent 思想史：四个时代的演进

> 📖 来源：`foundations/history-of-agent-ideas.mdx`

### 3.1 四阶段演进图

```
  符号系统          分布式智能         学习与强化         大规模预训练        现代智能体
  (Symbolic)   →   (Distributed)  →   (Learning)   →   (Pretraining)  →  (Modern Agent)
  ─────────        ────────────       ──────────        ────────────      ────────────
  专家系统          心智社会理论        连接主义+RL         LLM 装载先验知识    混合系统
  显式规则          多过程协调         从数据中学习         推理/规划/工具使用   结构化+学习
```

### 3.2 每个时代解决什么问题、暴露什么问题

| 时代 | 核心思想 | 解决 | 暴露 |
|------|---------|------|------|
| **符号系统** | 显式表示知识 | 窄领域可工作 | 常识缺失、脆弱、知识获取难 |
| **分布式智能** | 多个专门过程协调，非单一中心 | 从协作中涌现智能 | 协调成本高 |
| **学习系统** | 从数据和交互中学习，不完全预设 | 不必手工编码 | 推理难以检查 |
| **预训练** | 大规模装载语言与世界先验 | 推理、规划、工具使用更易获得 | 连贯但不一定正确 |

### 3.3 核心认知：现代 Agent 是混合体，不是纯粹体

四个时代的思想全部延续到当下：

```
符号思维  →  工具、模式、协议、确定性工作流规则
分布式思维 →  多智能体系统、模块化运行时
学习      →  模型权重、评估器、策略优化
预训练    →  在看到特定任务上下文之前的广泛先验知识
```

> 💡 实践目标不是选择某一个时代，而是**为真实任务组合正确的思想**。

## 4. LLM 基础：Agent 的"大脑"如何工作

> 📖 来源：`foundations/llm-foundations-for-agent-systems.mdx`

### 4.1 四个核心概念

| 概念 | 含义 | 对 Agent 设计的影响 |
|------|------|-------------------|
| **Next-Token Prediction** | 模型训练的是续写序列，不是符号证明 | 模型擅长模式压缩和语言控制，不擅长保证正确性 |
| **Tokenization & Embeddings** | 输入被分词并映射到向量空间 | 不是处理"原始人类文本"，理解有粒度限制 |
| **Transformer Attention** | 模型决定输入的哪些部分重要 | 注意力是有成本的，上下文越长越贵 |
| **Pretraining + Adaptation** | 广泛能力来自预训练，任务表现来自提示/工具/调优 | 基础模型 + 外部结构 = 可靠系统 |

### 4.2 Agent 视角：LLM ≠ 魔法

**常见错误认知 → 正确认知：**

| ❌ 错误 | ✅ 正确 |
|---------|--------|
| 模型记得比实际更多 | 模型知识有截止日期，需要外部检索 |
| 模型在做符号推理 | 模型主要受模式驱动，不是逻辑推演 |
| 上下文是免费资源 | 上下文是主要运行预算，要精心管理 |
| 更大的模型一定更好 | 成本、延迟和正确性需权衡 |

### 4.3 实用默认法则

```
模型 = 灵活的规划器和语言引擎

精确数据  →  外部工具
计算      →  外部工具
系统动作  →  外部工具

上下文限制 → 承认并围绕它设计，别假装不存在
```

### 4.4 LLM → Agent 的关键转变

```
LLM 单独 → 一次性文本生成

LLM + Environment + Loop → Agent

真正的差别：
  模型擅长"知道什么像好的回答"
  系统负责"确保回答是正确的、有来源的、可执行的"
```

## 5. Agent vs Workflow：谁拥有"下一步"

> 📖 来源：`foundations/agents-vs-workflows.mdx`

### 5.1 最清晰的区分方式

```
            Workflow                        Agent
            ────────                        ─────
         设计师拥有"下一步"              运行时拥有"下一步"
         脚本化路径 + 显式分支            根据状态、工具、目标选择动作
```

### 5.2 分别适用于什么场景

| 用 Workflow 当... | 用 Agent 当... |
|------------------|---------------|
| 路径已知 | 路径无法提前完全知道 |
| 规则稳定 | 系统需要搜索、探索、适应 |
| 可审计性 > 灵活性 | 任务运行期间环境可能变化 |
| 合规、审批、不可逆操作 | 有边界的探索和判断 |

### 5.3 实践中答案：混合体（Hybrid）

```
  ┌──────────┐
  │ Workflow │ ← 确定性步骤（审批、合规检查）
  │          │
  │    ↓     │
  │  Agent   │ ← 模糊步骤内部用 Agent 决策
  │    ↓     │
  │ Workflow │ ← 退出时再次进入确定性流程
  └──────────┘
```

> 💡 选错控制模型往往比选错模型更有破坏性。

## 6. Context Engineering：模型看到什么，比模型多强更重要

> 📖 来源：`systems/context-engineering.mdx`

### 6.1 核心命题

> "实际问题从来不是模型窗口有多大，而是**什么是最低限度但高信号、仍能让这一步成功的上下文**"

```
失败模式通常不是"模型很弱"，
而是"模型看到的指令、历史、证据和噪声组合不对"
```

### 6.2 上下文的组成

```
                    ┌─────────────────────┐
                    │    System Prompt    │ ← 长期稳定的系统指令
                    │     User Input      │ ← 当前用户输入
                    │     Memory          │ ← 跨步骤保留的状态
    Context =       │  Retrieved Knowledge│ ← 外部检索的证据
                    │    Scratchpad       │ ← 中间笔记/草稿
                    │    Tool Result      │ ← 工具调用返回
                    │    Environment      │ ← 环境信号
                    └─────────────────────┘
```

### 6.3 四个核心操作

| 操作 | 含义 | 典型场景 |
|------|------|---------|
| **Write** | 将状态持久化到即时提示词之外 | 保存决策、TODO、中间结果 |
| **Select** | 为下一步选择最相关的片段 | 从长历史中挑出对当前步骤有用的 |
| **Compress** | 总结或裁剪不再值得完整保真的内容 | 旧对话压缩为摘要 |
| **Isolate** | 防止不相关的工作互相污染 | 子 Agent 独立上下文窗口 |

### 6.4 四种上下文风险

| 风险 | 症状 | 对策 |
|------|------|------|
| **Context Poisoning** | 错误假设被保留并继续影响后续步骤 | 定期验证关键假设 |
| **Context Distraction** | 模型持续关注过去，而不是当前问题 | 压缩旧历史 |
| **Context Confusion** | 无关材料稀释任务 | Select 操作精准筛选 |
| **Context Clash** | 两部分信息冲突，模型锁定错误一方 | 优先级标记、冲突检测 |

### 6.5 设计目标：干净的状态传递

> 当一个系统能很好地支持长任务时，通常意味着它能**把从一个步骤到下一个步骤所需的状态精确交接过去**——不是最大程度的完整性，而是**干净的状态传递**。

## 7. Memory 与 Retrieval：Agent 的记忆体系

> 📖 来源：`patterns/agent-memory-and-retrieval.mdx`

### 7.1 三层记忆 + 两个外部层

```
  ┌─────────────────────────────────────────────────────────┐
  │                      工作记忆 (Working Memory)            │
  │              当前任务/会话的活动状态，容量和时间有限        │
  ├─────────────────────────────────────────────────────────┤
  │                      情景记忆 (Episodic Memory)           │
  │         有边界的事件、结果和经历，按语义+时间检索           │
  ├─────────────────────────────────────────────────────────┤
  │                     语义记忆 (Semantic Memory)            │
  │    跨任务的提炼事实、规则和关系，需要规范化+关系明确        │
  └─────────────────────────────────────────────────────────┘

  ┌──────────────────────┐    ┌──────────────────────┐
  │   检索层 (Retrieval)   │    │  笔记与工件 (Artifacts) │
  │  外部知识源，不属于     │    │  TODO、摘要、报告、    │
  │  Agent 自身            │    │  决策日志，可读存储    │
  └──────────────────────┘    └──────────────────────┘
```

### 7.2 所有权区分（关键设计原则）

| 类型 | 属于 | 生命周期 |
|------|------|---------|
| 会话记忆 | 正在运行的任务 | 会话结束即消失 |
| 持久记忆 | Agent 的运行历史 | 跨会话持久化 |
| 检索索引 | 外部文档/数据库/数据产品 | 由外部系统管理 |
| 笔记和工件 | 显式工作产物 | 人类可检查、编辑、批准 |

### 7.3 三个反复出现的设计选择

```
Promotion（提升）
  → 不是所有工作记忆项都应该变成持久内容
  → 需要判断哪些值得保留，哪些是噪声

Forgetting（遗忘）
  → 低价值或过时的记忆必须衰减、过期或归档
  → 错误记忆的代价很高：它们会反复返回

Boundary（边界）
  → 不是所有知识问题都是 RAG 问题
  → 检索回答"文档怎么说"≠ 替代任务记忆、决策日志
```

### 7.4 Memory vs Retrieval：什么时候用什么

| 场景 | 用 Memory | 用 Retrieval |
|------|----------|-------------|
| 跨任务保持连续性 | ✅ | ❌ |
| 跨决策跟踪历史动作 | ✅ | ❌ |
| 获取外部文档证据 | ❌ | ✅ |
| 证据在提问前留在提示词外 | ❌ | ✅ |
| 用 metadata filter 窄化范围 | ❌ | ✅ |
| 需要 citations 验证答案来源 | ❌ | ✅ |

### 7.5 实践中的混合检索

```
关键词匹配  →  精确实体和字面值
稠密检索    →  语义相似性
结构化过滤  →  时间范围、用户范围、文档类型

合适的系统通常把它们组合起来，而非只用一种
```

## 🧩 第一阶段知识体系总结

### 核心认知关系图

```
                Agent 是什么？
                （闭环决策系统）
                      │
        ┌─────────────┼─────────────┐
        │             │             │
   LLM 是大脑    环境+工具是身体   记忆是经验
   (思维引擎)    (行动能力)       (连续性)
        │             │             │
   Next-Token     四要素判定     三层记忆体系
   Prediction     Environment    Working
   Attention      Perception     Episodic
   Tokenization   Action         Semantic
   Adaptation     Autonomy       + Retrieval
        │             │             │
        └─────────────┼─────────────┘
                      │
              Context Engineering
              （模型看到什么）
               Write / Select
               Compress / Isolate
                      │
              ┌───────┴───────┐
              │               │
          Workflow          Agent
        （设计师决定）    （运行时决定）
              │               │
              └─── 混合系统 ───┘
```

### 路线图对照检查

| 路线图知识点 | 手册覆盖 | 状态 |
|-------------|---------|------|
| LLM 工作原理 (Token, Context Window, Attention, Function Calling) | `llm-foundations-for-agent-systems` | ✅ |
| Agent 生命周期 | `what-is-agent` + `the-agent-system` | ✅ |
| Context Engineering | `context-engineering` | ✅ |

| Agent vs Workflow | `agents-vs-workflows` | ✅ |
| Agent 思想史 | `history-of-agent-ideas` | ✅ |

### 第一阶段的三个最重要认知

1. **Agent = LLM + Environment + Loop**，不是装了 LLM 就叫 Agent
2. **Context Engineering > Prompt Engineering**，上下文管理比写提示词更重要
3. **混合体是答案**：纯 Workflow 太僵，纯 Agent 太不可控，混合结构是实践最优解
