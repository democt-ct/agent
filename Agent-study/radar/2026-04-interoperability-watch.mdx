## 摘要

2026年4月让互操作性层更容易拆分为不同的职责。A2A 进入第二年时，已具备可投入生产的 v1.0 和 Linux Foundation 治理。A2UI v0.9 将自己定位为可移植生成式界面的 UI 意图层。与此同时，MCP 继续深入传输、治理和企业需求，而不是收敛成一个包罗万象的单一协议。

## 为什么这很重要

智能体市场仍然把互操作性谈得像是某个单一标准会同时主导工具使用、智能体协同和生成式 UI。当前信号指向一个更好的方向。MCP 正在成熟为工具和传输的底层基座。A2A 正在强化外部智能体到智能体的边界。A2UI 则在尝试为现有前端上的界面意图表达建立标准。这样的拆分，比协议品牌之争更有价值。

## 证据与来源

- [协议与互操作性](/zh-Hans/systems/protocols-and-interoperability)：用于保持 MCP 和 A2A 边界清晰的仓库原生对比页面。
- [A year of open collaboration: Celebrating the anniversary of A2A](https://opensource.googleblog.com/2026/04/a-year-of-open-collaboration-celebrating-the-anniversary-of-a2a.html)：Google 将 A2A 的第一年概括为 Linux Foundation 治理、2026 年 3 月的 v1.0 发布，以及更广泛的生态支持。
- [A2UI v0.9: The New Standard for Portable, Framework-Agnostic Generative UI](https://developers.googleblog.com/a2ui-v0-9-generative-ui/)：Google 将 A2UI 定位为一种与框架无关的方式，让本地或远程智能体能够针对现有组件目录声明 UI 意图。
- [The 2026 MCP Roadmap](https://blog.modelcontextprotocol.io/posts/2026-mcp-roadmap/)：MCP 维护者将这一年的重点放在传输可扩展性、智能体通信、治理成熟度和企业就绪性上。

## 需要关注的信号

- 随着更多厂商将智能体到智能体交换引入生产环境，A2A 是否能继续保持与 MCP 的互补边界清晰。
- A2UI 是否能超越早期的 Google 相关实验获得采用，尤其是通过 React 和社区渲染器。
- MCP 的智能体通信工作最终会不会缩小工具访问与智能体间交换之间的区别，还是会让这种区别更清晰。
