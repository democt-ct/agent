# 09-推理优化技术栈

> 学完这篇要知道：推理为什么慢、显存带宽才是瓶颈、FlashAttention/vLLM/量化/speculative decoding 怎么把效率提 10 倍。

---

## 第一原理：瓶颈不在计算，在搬运

生成阶段（decode）每吐一个 token：读 14GB 权重（7B 模型 fp16）从 HBM → SRAM → 算 1 次 forward → 输出 1 个 token。GPU 算力/显存带宽的理论平衡点是 156 FLOP/byte，但 decode 实际计算密度只有 2 FLOP/byte——**GPU 99% 时间在等数据搬完**。

所以几乎所有推理优化都在干一件事：**减少要搬的数据量**。

---

## 优化技术栈（从底层到上层）

### 1. FlashAttention → 长序列快 2-4 倍

标准 attention 要把完整的 (seq, seq) 矩阵写入显存，长序列时这步 IO 是绝对瓶颈。FlashAttention 把 Q/K/V 切成小块，在 SRAM (静态随机存取存储器)里增量算 softmax，永远不把完整 attention 矩阵写入 HBM。显存从 O(n²) → O(n)。

**今天所有框架都默认开，你不用管——但面试要能讲清楚 IO-aware 的思想（输入输出感知，不仅仅关注理论上的计算复杂度，还必须将数据存储层级之间的移动成本作为核心考量因素）。**

### 2. GQA（分组查询注意力）→ KV Cache ÷4

标准 MHA 每个 head 有独立 K/V。GQA 把 head 分 G 组，组内共享 K/V。Llama/Qwen/Mistral 全用 GQA（G=8），KV Cache 省 4-8 倍，效果几乎不掉。

### 3. 量化（GPTQ/AWQ）→ 显存 ÷4、速度 ×2 模型压缩技术

把 fp16 权重压成 4 bit（int4）。AWQ 的 trick：**保留前 1% 重要权重的精度，其余激进量化**——效果几乎无损。

- fp16 → int8：大小减半，几乎无损
- int8 → int4：大小再减半，掉 1-3% 精度
- NF4（QLoRA 格式）：非均匀量化，掉<1%

**生产标配**：4-bit AWQ 量化 + vLLM 部署。

### 4. Speculative Decoding → 快 2-3 倍

小模型（draft）先猜 4-6 个 token，大模型一次性 verify。数学上零精度损失（rejection sampling 保证）。greedy 时加速最明显，高温采样时效果减半。

### 5. PagedAttention（vLLM 核心）→ 并发 ×4

传统 KV Cache 连续分配，但实际生成长度远小于预留值，显存浪费 60-80%。vLLM 把 KV Cache 切成 16 token 的页，按需分配——模仿操作系统虚拟内存。同等显存跑 2-4 倍并发。

### 6. Continuous Batching → 吞吐 ×5-10

每个 decode step 动态重组 batch——长请求不阻塞短请求。vLLM/TGI/SGLang 都默认开。

### 7. Prefix Caching → system prompt 只算一次

共享前缀（如长 system prompt）的 KV Cache 直接复用，多个请求只算一次 prefill。节省 30-50% 延迟。

## 优化栈速查

| 技术                 | 效果                 | 代价             |
| -------------------- | -------------------- | ---------------- |
| GQA                  | KV Cache ÷4-8       | 训练时改架构     |
| FlashAttention       | 长序列 2-4× 加速    | 已是默认，无代价 |
| INT4 量化            | 显存 1/4、速度 2-3× | 1-3% 精度损失    |
| PagedAttention       | 显存利用率 90%+      | 引擎层实现       |
| Continuous Batching  | 吞吐 5-10×          | 引擎层实现       |
| Speculative Decoding | 单请求 2-3×         | 需 draft 模型    |
| Prefix Caching       | 共享前缀场景 2-5×   | 额外显存         |

## 生产部署速成

```bash
# vLLM 部署 Qwen2.5-7B，一行搞定
python -m vllm.entrypoints.openai.api_server \
  --model Qwen/Qwen2.5-7B-Instruct \
  --enable-prefix-caching \
  --gpu-memory-utilization 0.9
```

单卡 A100 能跑 1500-3000 token/s 吞吐——比朴素 HuggingFace 快 10 倍。

---

## 容易踩的坑

1. **盲目堆 batch_size** —— 没用，换 continuous batching
2. **量化"免费午餐"** —— 代码/推理/长上下文场景可能掉 5-10%，先测再上
3. **ignore prefill 延迟** —— TTFT = prefill + 第一个 token，长 prompt prefill 秒级
4. **fp16 而非 bf16** —— A100+ 用 bf16 更稳定（动态范围大，不溢出）
5. **speculative decoding 当万能药** —— 高温采样（创意写作）接受率低反而慢
