# 05-上下文窗口

> 学完这篇要知道：context window 不是越长越好——RoPE 怎么做到的、长上下文有什么代价、什么时候必须用超长窗口。

---

## context window 到底是什么

模型一次能"看见"的 token 数量上限。GPT-4 训练时如果 context=8192，那它最多记住前面 8191 个 token 的内容。第 8192 个 token 进来，第 1 个 token 就"遗忘"了（被截断或 attention mask 丢弃）。

**实际影响你的**：RAG 塞了 5000 token 资料 + 3000 token 对话历史 → 还剩多少给 prompt 指令和输出？算清楚这个才有空间意识。

---

## 怎么做到长上下文：RoPE

训练时模型只见过 4K token。推理时凭什么能看 128K？**RoPE（旋转位置编码）**。

核心 trick：Q 和 K 被旋转一个角度，角度 = 位置 / θ。两个位置的距离 = 旋转角度之差。模型学的不是"第 128 个位置该怎样"，而是"两个位置差 50 步该怎样"——**相对位置**。

外推原理：只要 θ 设置得当，模型在"差 1000 步"上学到的模式能泛化到"差 10000 步"。Llama 3 把 θ 从 10000 提到了 500000，直接让外推范围扩大 50 倍。

---

## 长上下文的真正代价

| 代价 | 具体数字 | 为什么 |
|------|---------|--------|
| **KV Cache 显存** | 每 token 约 128KB（7B 模型） | 128K token → 16GB 显存只装 KV Cache |
| **Prefill 延迟** | 10K token 约 5-10 秒（A100） | O(n²) 的 attention 矩阵计算 |
| **注意力衰减** | 中间信息容易丢 | 模型对开头和结尾的东西记得最牢 |
| **成本** | 输入按 token 计费 | 128K context 一次调用的输入费 = 128 倍 |

实际使用中，**1M token context ≠ 1M token 的有效记忆**。Gemini 2.5 Pro 号称 2M，但评测显示超过 200K 后准确度就开始掉。"大海捞针"测试只是下限——在 1M token 里精确回答针对中间某一段的问题，不代表能综合处理整本书。

---

## 什么时候必须用长上下文

| 场景 | 大概要多少 | 备注 |
|------|-----------|------|
| 日常聊天 | 4K-8K | 存 10 轮对话就够了 |
| RAG 单次检索 | 2K-4K | 资料片段 + 指令 |
| 代码项目理解 | 32K-128K | 整个 repo 上下文 |
| 长文档分析 | 128K+ | 论文、合同、书籍 |
| 多轮 Agent 会话 | 32K-128K | 工具调用历史很长 |

**反常识**：大部分 Agent 任务不需要超长 context——依赖 RAG 比硬塞进 context 更准更便宜。

---

## 容易踩的坑

1. **context 越长 ≠ 效果越好** —— 超过有效范围后信息越多越乱，模型会"迷失在中间"
2. **只关注 context limit 不关注 prefill 延迟** —— 100K token 怼进去，用户等 30 秒
3. **用 context 替代 RAG** —— context 里的信息没有源文件可追溯，幻觉无证据
4. **忘了各家 context 是真 128K 还是"声称 128K"** —— Claude 200K 真实可用，DeepSeek-V3 官方标注 128K 但长尾实际掉精度
5. **system prompt 塞太多东西** —— 虽然后面不会被逐 token 计费，但每次 prefill 都重算
