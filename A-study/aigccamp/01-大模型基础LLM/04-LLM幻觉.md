# 04-LLM 幻觉：为什么胡说八道

> 学完这篇要知道：幻觉分几种、用什么手段让模型少编、为什么永远不可能 100% 消除。

---

## 幻觉不是 bug，是 LLM 工作原理的副产品

模型不是查数据库，是"根据上下文猜下一个词"。当它不确定时就按概率最高的路径接着编——这在人类看来是"胡说"，在模型视角是"最合理的续写"。

---

## 幻觉的三类

| 类型 | 例子 | 根因 |
|------|------|------|
| **事实错误** | "李白生于公元 701 年"（实际 701 年）→ 有时对有时错，不稳定 | 训练数据冲突 / 记忆模糊 |
| **逻辑矛盾** | 前面说"A>B"，后面说"B>A" | 长上下文注意力衰减 |
| **虚构编造** | "据《Nature》2023 年发表的研究..."（论文不存在） | 语言模式匹配，不是查证 |

**最容易出现幻觉的场景**：冷门知识、精确数字/日期、引用来源、长链条推理的中间步骤。

---

## 减少幻觉的实操手段

### 1. RAG（检索增强生成）→ 最有效

答案不在模型参数里搜，而是从外部文档检索 → 注入 context → 只基于 context 回答。好处：来源可追溯、实时可更新。代价：多一次检索延迟。

### 2. Temperature 调低 → 简单直接

temp=0.1 时模型倾向于选最确信的 token，乱编的概率大降。代价：回答变保守。

### 3. 提示词加约束 → 零成本

"如果你不确定，就说不知道"、"只基于以下资料回答"、"列出你的信息来源"——这些话会显著改变模型行为。

### 4. Self-Consistency（自洽性）→ 推理题用

让模型用不同思路做 5 次，少数服从多数。数学推理上能提 10-20 个点。代价：5 倍成本。

### 5. Grounding（事实核查）→ 生产必备

生成后跑一遍事实核查——对齐来源、验证数字、检查一致性。Google、必应 API 都能做自动化 grounding。这是企业级 RAG 产品的标配。

---

## 为什么不可能 100% 消除

- 模型没有"不知道"的概念——它只有概率分布，永远会说点什么
- 有些知识本身就在训练语料里冲突（不同来源说法不同）
- 长文本中前后一致性是有复杂度下限的

**应对哲学**：不求消灭幻觉，求控制在可接受范围 + 有兜底机制（给用户提示 + 可溯源）。

---

## 快速诊断表

| 现象 | 可能原因 | 怎么试 |
|------|---------|--------|
| 回回编论文引用 | 没开 RAG / 领域太窄 | 加资料库 + RAG |
| 数字总错 | temp 太高 | temp → 0.1 |
| 前后矛盾 | context 太长 | 缩短 / 分块处理 |
| 重复某个幻觉 | prompt 里没禁止 | 加"不确定就说不知道" |
| 逻辑推理翻车 | 中间步骤太长 | 用 CoT + self-consistency |
