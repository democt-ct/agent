# 07-Prompt 工程基础

> 学完这篇要知道：好的 prompt 为什么有效、结构化的 prompt 怎么写、System/User/Assistant 三个角色怎么分工。

---

## Prompt 不是"说话客气点"——是控制模型行为的唯一杠杆

对 LLM 来说，prompt 是它唯一能看到的"任务描述"。同样的模型，prompt 好坏决定效果上下十倍差距。

---

## System / User / Assistant 三角色

| 角色 | 什么时候用 | 放什么 |
|------|-----------|--------|
| **System** | 整个对话的设置 | 角色定义、规则约束、输出格式要求 |
| **User** | 每一轮的提问 | 具体任务、上下文、数据 |
| **Assistant** | 模型之前的回答（few-shot） | 示例答案 |

**关键事实**：System 的权重通常高于 User——模型更倾向于遵守 system prompt 的约束。所以"身份设定"、"禁止事项"放 system 是最优解。

---

## 结构化 Prompt 模板

好 prompt 通常包含五要素：

```
1. 角色：你是一个 XX 专家
2. 任务：把下面这段话翻译成英文
3. 约束：保持语气、不要遗漏信息、不确定写 [UNCERTAIN]
4. 格式：输出 JSON，key 为 "original" 和 "translated"
5. 输入/示例：（接着给具体内容）
```

**为什么结构化有效**：模型是在"续写"——给它某种格式的输入，它就会以那种格式续写。你给混乱的 prompt，它输出的结构也混乱。

---

## 写 Prompt 的六个技巧

| 技巧 | 做法 | 为什么有效 |
|------|------|-----------|
| **角色设定** | "你是资深 Python 工程师" | 激发模型内对应领域的知识和风格 |
| **正反例** | "好的回答是这样的...不要那样..." | 比纯文字描述直观 10 倍 |
| **Chain of Thought** | "请一步步思考" | 强制模型展开推理而非直觉回答 |
| **结构化输出** | "输出 JSON 格式" | 框死输出，防止跑偏 |
| **负面约束** | "不要使用复杂术语" | 模型的 subtraction 是靠 prompt 引导 |
| **Few-shot** | 给 2-3 个示例 | prompt 里的示例 = 模型最直接的学习信号 |

---

## CoT（Chain of Thought）：让模型"多想一步"

**不用 CoT**："15 × 17 = ?"→ 可能直接猜 260（错了）

**用 CoT**："请逐步计算：15 × 17。首先..."

模型会展开：
```
15 × 10 = 150
15 × 7 = 105
150 + 105 = 255
```

为什么有效？因为**写出来中间步骤变相增加了"计算深度"**——原本 1 次 forward 猜答案，现在 N 次 forward 每次只做一小步。每一步的概率分布比直接猜最终答案靠谱。

**CoT 变种**：
- **Zero-shot CoT**：加一句"Let's think step by step"（提升 10-30% 推理准确率）
- **Few-shot CoT**：给 2-3 个"问题→思维链→答案"示例
- **Self-Consistency**：跑 5 条不同 CoT，投票选多数

---

## Few-shot 示例怎么选

1. **选和当前任务格式一致的** —— 不是内容相似，是结构相似
2. **覆盖边界 case** —— 示例里要包含"简单/正常/棘手"各一个
3. **数量**：2-3 个通常最佳，超过 5 个收益递减
4. **顺序**：简单→正常→棘手，让模型从易到难学

---

## 生产 Prompt 的几个铁律

1. **Prompt 也是代码，要版本管理** —— 存在 Git 里，别散落在聊天记录
2. **上线前用 eval 集测** —— 不要凭感觉"这个 prompt 挺顺的"
3. **System prompt 变化是 A/B test** —— 不是"我觉得改了更好"，要量化验证
4. **用户输入永远要做注入防护** —— prompt injection 是 Agent 安全第一课
5. **长 system prompt 不等于好 system prompt** —— 每多一行，模型遵循度就减一分

---

## 容易踩的坑

1. **system prompt 写太长** — 100 条规则 = 0 条被遵守，精简到 5 条核心规则最稳
2. **"不准说 X"反而让模型更容易说 X** — 负面指令会激活相关语义，试试改成正向约束
3. **示例选得太好** — 模型学不会处理"不完美的真实输入"
4. **中文 prompt 中英混杂** — 切词不一致可能导致忽视英文部分
5. **改 prompt 不改 eval** — 你以为更好了，可能只是换个花样犯错
