# 患者过敏安全机制设计

## 问题场景

### 场景 A：已知过敏与知识冲突

面试官提出的典型场景：

> 患者说"我头疼"，患者档案中记录为"阿司匹林过敏"，但 RAG 知识库中推荐头疼的标准治疗是阿司匹林。模型应该怎么回复？

**核心矛盾**：患者隐私数据（过敏史）与公共医学知识（标准治疗）冲突时，系统有没有安全防护机制？

### 场景 B：未知过敏

一个更隐蔽的危险场景：

> 患者说"我头疼"，但患者档案中**未记录**过敏史（`allergy_history` 为空或未填写）。此时 RAG 知识库推荐阿司匹林，系统不知道患者是否过敏，LLM 在没有任何阻碍的情况下自信推荐阿司匹林。

**核心矛盾**：信息缺失本身也是一种风险——"不知道过敏"不等于"没有过敏"。

---

## 现状缺陷

修改前的系统存在三个层面的安全缺口：

1. **Prompt 层**：MEDICAL SAFETY 指令中没有任何关于"检查患者过敏史后再推荐药物"的硬约束
2. **RAG 层**：知识检索时完全不感知患者过敏信息，可能返回含过敏药物的知识块
3. **输出层**：LLM 生成回答后没有程序化校验，纯靠 LLM 自觉

患者过敏信息虽然存在于 `Patient.allergy_history` 字段中，且在回答 Prompt 中可见，但当上下文较长或知识块权威感强时，LLM 可能忽略过敏提示，直接推荐标准治疗。

---

## 设计原则

```
患者过敏史（绝对硬约束）
    ↓ 高于
RAG 知识库（参考信息）
    ↓ 高于  
LLM 自身知识（通用医学常识）
```

三层防护，层层递进，确保过敏信息不被遗漏。

---

## 实现方案

### 第一层：过敏药物提取与传递

**位置**：`app/api/mcp_routes.py`

**新增函数**：`_parse_allergy_drugs(allergy_history: str) -> list[str]`

从患者 `allergy_history` 字段中解析药物名称，支持两种匹配策略：
1. 基于预定义药物词典（`DRUG_TERMS` + `DRUG_SUFFIXES`）的精确匹配
2. 正则提取"对XXX过敏"模式中的药物名

**调用时机**：在 `mcp_agent_query` 和 `mcp_agent_query_with_image` 中，身份解析完成后立即提取过敏药物列表，并通过 `allergy_drugs` 参数传递给后续所有回答生成环节。

---

### 第二层：RAG 知识过滤

**位置**：`app/api/mcp_routes.py`

**新增函数**：`_filter_allergy_from_knowledge(context: str, allergy_drugs: list[str]) -> str`

在 `_build_knowledge_context_block` 中，检索到知识块后逐行扫描，剔除任何包含患者过敏药物的行。

**过滤策略**：大小写不敏感的逐行扫描，过敏药物名出现在知识块的任何位置（标题、正文、标签）即剔除整行。

---

### 第三层：Prompt 硬指令注入

**位置**：`app/mcp/llm_router.py`

在 `_generate_open_answer_v2` 和 `_build_answer_prompt` 两个回答生成 Prompt 中注入过敏安全指令：

**英文 Prompt（Open Answer V2）**：
```
MEDICAL SAFETY:
- Always check patient allergy history before mentioning any drug

PATIENT ALLERGY WARNING: The patient is allergic to: 阿司匹林.
NEVER recommend or suggest these drugs or any drugs in the same class.
If a standard treatment would normally involve any of these drugs,
clearly state that the patient cannot use them due to allergy and suggest alternatives.
```

**中文 Prompt（Tool Answer）**：
```
患者过敏警告：患者对 阿司匹林 过敏。
绝不要推荐这些药物或同类药物。
如果标准治疗包含其中任何药物，必须明确说明患者不能使用并建议替代方案。
```

**逻辑**：当 `allergy_drugs` 非空时注入该块；为空时不注入，避免 Prompt 膨胀。

---

### 第四层：回答后安全扫描

**位置**：`app/mcp/llm_router.py`

**新增函数**：`_check_allergy_safety(answer: str, allergy_drugs: list[str]) -> str`

在 LLM 生成回答后，扫描回答文本中是否出现患者过敏药物。命中时追加醒目警告：

```
---
⚠️ **安全提醒**：系统检测到回答中提到了 **阿司匹林**。
根据患者档案，患者对该药物**已知过敏**。
请在用药前务必确认患者的过敏史，避免使用过敏药物。
```

**调用位置**：三个回答生成路径的返回前各插入一次调用：
1. `_generate_open_answer_v2` — 直接回答路径
2. `_generate_answer_text` — 工具查询路径
3. 均位于 `_invoke_text_prompt` 之后、`return` 之前

---

### 第五层：未知过敏免责注入（针对场景 B）

**要解决的问题**：当患者档案未记录过敏史（`allergy_history` 为空或未填写）时，系统不知道患者是否对任何药物过敏。此时即使前三层全部空转，仍需提醒 LLM 不要在未确认过敏史的情况下自信推荐药物。

**位置**：`app/api/mcp_routes.py` + `app/mcp/llm_router.py`

**检测逻辑**（mcp_routes.py）：

```python
allergy_drugs: list[str] = []
allergy_history_unknown: bool = False
if resolved_patient_id:
    patient = get_patient(db, resolved_patient_id)
    allergy_history = (patient.allergy_history or "").strip()
    if allergy_history:
        allergy_drugs = _parse_allergy_drugs(allergy_history)
    else:
        allergy_history_unknown = True  # ← 档案中无过敏记录
```

**信号传递**：`allergy_history_unknown` 与 `allergy_drugs` 并列传递，贯穿 `run_agent_tool_query` → `_generate_open_answer_v2` / `_build_answer_prompt` → `_generate_answer_text` → `_select_direct_path`。

**Prompt 注入**：

当 `allergy_history_unknown = True` 时，两个回答生成 Prompt 分别注入行为指令，引导 LLM 给出全面、适当的回答，而不是输出一个独立的提醒块：

英文 Prompt（Open Answer V2）：
```
When your response includes medication-related advice,
make sure the answer is comprehensive and appropriate — naturally include
necessary precautions (e.g., consult a doctor, check contraindications)
as part of the response flow, not as a separate warning block.
```

中文 Prompt（Tool Answer）：
```
回答涉及药物建议时，请给出全面、适当的回答，
自然包含必要的用药提醒（如咨询医生、确认禁忌等）
作为回答行文的一部分，不要以独立提醒块的形式出现。
```

**与过敏警告的关系**：两者互斥但共存于同一 Prompt 模板中：

```
{allergy_warning}          ← 已知过敏时注入硬警告（场景A）
{allergy_unknown_disclaimer} ← 未知过敏时注入"全面回应"行为指令（场景B）
```

当 `allergy_history_unknown = False` 且 `allergy_drugs` 非空时 → 场景 A，注入硬警告。
当 `allergy_history_unknown = True` 且 `allergy_drugs` 为空时 → 场景 B，注入自然融入的行为指令。
当 `allergy_history_unknown = False` 且 `allergy_drugs` 为空时 → 无患者绑定（general 模式），两者均不注入。

## 数据流路径

```
mcp_agent_query
  │
  ├─ _resolve_bound_identity → patient_id
  │
  ├─ get_patient(patient_id) → patient.allergy_history
  │
  ├─ _parse_allergy_drugs(patient.allergy_history) → allergy_drugs = ["阿司匹林"]
  │
  ├─ _build_knowledge_context_block(question, hospital_id, allergy_drugs)
  │     │
  │     └─ build_knowledge_context(db, ...) → ["头痛可服用阿司匹林", "头痛可冷敷..."]
  │     └─ _filter_allergy_from_knowledge(context, ["阿司匹林"]) → ["头痛可冷敷..."]
  │
  └─ run_agent_tool_query(..., allergy_drugs=["阿司匹林"])
        │
        ├─ _generate_open_answer_v2(llm, ..., allergy_drugs=["阿司匹林"])
        │     │
        │     ├─ Prompt 注入: "PATIENT ALLERGY WARNING: ...阿司匹林..."
        │     ├─ LLM 生成回答
        │     └─ _check_allergy_safety(answer, ["阿司匹林"])  ← 安全扫描
        │
        └─ _generate_answer_text(llm, ..., allergy_drugs=["阿司匹林"])
              │
              ├─ _build_answer_prompt(..., allergy_drugs=["阿司匹林"])
              │     └─ Prompt 注入: "患者过敏警告：患者对 阿司匹林 过敏..."
              ├─ LLM 生成回答
              └─ _check_allergy_safety(answer, ["阿司匹林"])  ← 安全扫描
```

## 边界情况

| 情况 | 行为 |
|------|------|
| 患者有已知过敏史（场景 A） | `allergy_drugs` 非空 → RAG 过滤 + Prompt 硬警告 + 输出扫描，三层全开 |
| 患者过敏史未记录（场景 B） | `allergy_drugs` = `[]`，但 `allergy_history_unknown` = `True` → Prompt 注入自然融入的行为指令："在行文中提醒患者确认过敏情况" |
| 患者无绑定（`general` 模式） | `allergy_drugs` = `[]`，`allergy_history_unknown` = `False` → 两者均不注入，无副作用 |
| RAG 知识块部分包含、部分不包含过敏药物 | 过滤后仅保留安全条目；若全部被过滤，返回 `None` |
| LLM 在回答中提到了过敏药物的同义词或商品名 | 当前仅字面匹配，同义词可能漏过（后续可扩展模糊匹配） |
| 患者过敏药物不在已知词典中 | `_parse_allergy_drugs` 的正则回退可捕获"对XX过敏"模式的药物名 |

---

## 设计局限与后续扩展

1. **药物同义词匹配**：当前为精确字面匹配。"阿司匹林"与"乙酰水杨酸"、"拜阿司匹林"可能不匹配。后续可引入药物同义词库。
2. **药物类别推理**：如患者对阿司匹林过敏，但 LLM 推荐了布洛芬（同为 NSAID），当前不会触发警告。后续可引入药物分类层级（ATC/WHO 分类）。
3. **过敏严重程度分级**：当前所有过敏等同对待。后续可在 `allergy_history` 中结构化存储过敏类型和严重程度（如过敏性休克 vs 轻度皮疹）。
4. **未知过敏免责的适用面**：当前仅在患者已绑定但 `allergy_history` 为空时注入免责声明。对于匿名用户（general 模式），不注入免责声明（因为无身份可关联），这符合"无身份则不提示"的语义。但如果未来引入"半匿名"模式（如仅知道手机号但不知完整档案），需重新审视此逻辑。
