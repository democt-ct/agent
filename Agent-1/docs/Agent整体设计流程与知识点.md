# 患者智能辅助 Agent — 整体设计流程与知识点

## 前言

本文档描述一个面向医疗场景的患者智能辅助 Agent 的**完整系统设计**。它不是单纯的对话机器人，而是一个集成了身份认证、多层记忆、结构化数据查询、知识检索、多模态理解和语音输出的智能体系统。以下从**整体流程**和**知识点**两个维度展开。

---

### 两种模式

系统支持两种对话模式：

| 模式 | 适用场景 | 行为差异 |
|------|---------|---------|
| **general** | 普通问答 | 不写入短期消息到 DB，只维护内存中的工作记忆 |
| **memory** | 需要持久化上下文 | 每次用户消息写入 `memory_conversation_messages` 表，支持跨会话回溯 |

## 第二部分：各层关键知识点

### 知识点 1：身份认证与解析层

**要解决的问题**：在医疗场景中，所有患者数据必须绑定到正确的人。系统必须在答复前确定"谁在问"。

**三种身份确认方式**：

| 方式 | 来源 | 优先级 | 说明 |
|------|------|--------|------|
| Auth Token | 显式传入 | 最高 | HMAC-SHA256 签名的 token，含 patient_id + hospital_id + 过期时间 |
| 自然语言提取 | 对话文本 | 中 | 从"我是张三，1958年出生"中自动提取姓名、出生年、手机号 |
| 直接传参 | API 参数 | 低 | 直接传 patient_id，适用于测试场景 |

**Auth Token 原理**：
- 载荷：`{patient_id, hospital_id, iat, exp}`
- 签名：HMAC-SHA256(secret, JSON 载荷)
- 格式：`base64url(payload).hex(signature)`
- 签发后返回给客户端，后续请求带上即可免去身份解析

**身份解析的回退策略**：当没有 token 时，系统从对话文本中提取身份线索（姓名、出生年、手机号），与患者库模糊匹配。精确命中一人则自动绑定；匹配多人则标记为候选，等待后续对话进一步确认。

---

### 知识点 2：短期工作记忆

**要解决的问题**：单次对话的上下文不能无限增长，需要结构化压缩。

**四个组件**：

| 组件 | 内容 | 用途 |
|------|------|------|
| **Recent Messages** | 最近 5 轮 user + assistant 消息 | 保持局部对话连贯性 |
| **Session State** | 意图、当前话题、目标、已确认事实、未决问题、身份状态 | 让 LLM 理解对话进度 |
| **Active Entities** | 当前活跃的药物、症状、检查、指标 | 快速聚焦当前医疗实体 |
| **Risk Signals** | 红旗信号（胸痛/晕厥）、用药信号、监测信号 | 触发安全提醒 |

**更新策略**：每次新消息到来时，系统从消息中提取实体（基于关键词匹配），更新活跃实体列表，同时根据对话内容推断新的会话状态。

---

### 知识点 3：长期记忆系统

**要解决的问题**：跨会话的医疗知识需要被持久化、分层管理，且不能与短期对话混淆。

**详细结构**：

```
短期对话消息 ──→ 关键事件 ──→ 画像
  (会话级)        (长期知识)    (聚合摘要)
```

**三层画像**：

| 画像名称 | 数据来源 | 用途 | 更新触发 |
|---------|---------|------|---------|
| 业务画像(BusinessProfile) | 病历 + 就诊记录 | 医生工作站、AI 医学推理 | 病历/就诊更新时 |
| 对话画像(ConversationProfile) | 对话消息 | AI 沟通风格适配 | 每次对话抽取 |
| 融合画像(UserProfile) | 两者合并 | AI 整体患者认知 | 定期合并更新 |

**关键事件（Key Events）**：从原始数据中提炼的"有长期价值的知识点"。每条事件记录：类型、内容摘要、影响、置信度、优先级、来源证据。支持幂等合并（相同 canonical_key 的事件合并而非重复）。

**偏好（Preferences）**：独立于抽取流水线，是患者或医生的显式配置。包含回答风格、术语级别、风险提醒强度等。

---

### 知识点 4：知识检索（RAG）

**要解决的问题**：AI 需要引用外部医学知识来支撑回答，而不能仅靠训练数据。

**混合检索策略**（四维加权）：

```
final_score = 0.58 × vector_score
            + 0.22 × keyword_score
            + 0.12 × metadata_score
            + 0.08 × recency_score
            + 0.04 × confidence
```

| 维度 | 含义 | 实现方式 |
|------|------|---------|
| **向量分数** | 语义相似度 | 文本嵌入 → 余弦相似度（支持 OpenAI / HuggingFace / 本地 Hash） |
| **关键词分数** | 字面匹配 | 分词后 Jaccard 式重叠度 |
| **元数据分数** | 医院匹配 / 领域匹配 | 精确过滤（hospital_id = ? / domain = ?） |
| **时效分数** | 知识新鲜度 | 指数衰减：7 天内 = 1.0，30 天 = 0.8，180 天 = 0.5 |

**嵌入层降级策略**：
1. 优先用 OpenAI Embeddings（需 API Key）
2. 次选 HuggingFace 本地模型（如 `BAAI/bge-small-zh-v1.5`）
3. 最后用 HashTextEmbedder（确定性哈希，不依赖外部模型）

**知识版本控制**：每条知识块可设置 `effective_from`（生效时间）和 `expires_at`（过期时间），过期知识自动不参与检索。

---

### 知识点 5：意图识别与规划

**要解决的问题**：用户的自然语言问题需要被映射到具体的系统动作。

**采用"模板优先 + LLM 兜底"的分层策略**：

```
用户问题
    │
    ├─ 命中关键词模板？ → 直接返回意图（速度快，确定性高）
    │   (如"就诊"→ visit_records_query, "病历"→ medical_records_query)
    │
    └─ 未命中？ → 调用 LLM 分类（慢但覆盖广）
                  (含 intent + confidence + focus)
```

**预定义意图集合**：

| 意图 | 触发关键词 | 对应工具序列 |
|------|-----------|-------------|
| `visit_records_query` | 就诊/挂号/医生 | `get_visit_records` |
| `medical_records_query` | 病历/诊断/治疗 | `get_medical_records` |
| `symptom_consultation` | 头晕/胸闷/咳嗽 | `get_patient_profile` |
| `patient_profile_summary` | 画像/总结/概况 | `get_patient_profile` |
| `multimodal_summary` | 图片输入 | `get_patient_profile` |
| `general_medical_question` | 其他 | 无工具，纯文本回答 |

**计划生成**：每个意图映射到 1-2 个工具调用步骤。对复杂情况，LLM 可生成多候选计划，系统通过**共识投票**（Consensus Voting）选择执行路径。

---

### 知识点 6：MCP 工具系统

**要解决的问题**：LLM 不能直接操作数据库，需要通过注册好的工具接口来获取数据。

**注册的 5 个核心工具**：

| 工具名 | 功能 | 身份保护 |
|--------|------|---------|
| `issue_identity_token` | 签发测试用身份 token | 仅测试环境可用 |
| `verify_identity` | 校验 token 有效性 | 返回身份上下文 |
| `get_medical_records` | 查询患者病历 | 需 patient_id 或 auth_token |
| `get_visit_records` | 查询就诊记录 | 需 patient_id 或 auth_token |
| `get_patient_profile` | 聚合返回患者所有信息 | 需 patient_id 或 auth_token |

**设计要点**：
- 工具参数统一通过 `patient_id` 或 `auth_token` 二选一进行身份校验
- 每个工具内部都经过 `_resolve_patient_context` 校验身份一致性
- 返回数据自动序列化（SQLAlchemy 模型 → dict）

---

### 知识点 7：身份校验与数据安全

**要解决的问题**：患者数据是敏感隐私信息，必须确保操作者确实是被授权的人。

**双因子校验链**：

```
用户传入 auth_token / patient_id + hospital_id
    │
    ▼
验证 token 签名（HMAC-SHA256）
    │
    ▼
验证 token 未过期（exp 字段）
    │
    ▼
查询患者是否存在且 is_active = True
    │
    ▼
验证传入的 hospital_id 与患者记录一致
    │
    ▼
通过 → 返回患者身份 + 授权上下文
```

**约束**：
- token 中的 patient_id 和传入的 patient_id 必须一致（否则 403）
- token 中的 hospital_id 和患者记录的 hospital_id 必须一致（否则 403）
- 匿名会话（未确认身份）的消息写入单独的缓冲区表，不与患者关联

---

### 知识点 8：多模态理解（Vision）

**要解决的问题**：患者有时会上传图片（检查报告照片、药品包装、患处照片），AI 需要理解图片内容。

**流程**：
1. 用户提交问题 + 图片
2. 图片 Base64 编码后传给视觉 LLM（支持 OpenAI 兼容接口）
3. 视觉 LLM 返回图片文字摘要
4. 该摘要作为 `image_analysis` 字段注入到后续的问答上下文

**模型回退**：如果首选视觉模型（`VISION_MODEL`）调用失败，自动回退到次选模型（`OPENAI_VISION_MODEL`），最后到通用模型。若全部失败则抛出可读错误。

---

### 知识点 9：语音输出（TTS）

**要解决的问题**：对于语音交互场景（如手机端、音箱端），需要将文字回答转为语音。

**双引擎支持**：

| 引擎 | 适用场景 | 配置方式 |
|------|---------|---------|
| **OpenAI Compatible TTS** | 远程 API，音质高 | `OPENAI_TTS_MODEL` + `OPENAI_API_KEY` |
| **Kokoro** | 本地运行，隐私好 | `KOKORO_REPO_ID` 指向本地模型文件夹或 HF cache |

**自动降级**：配置了 OpenAI TTS 但网络失败时，如果 Kokoro 可用则自动切换到 Kokoro。

---

### 知识点 10：Prompt 组装策略

**要解决的问题**：AI 的最终回答质量取决于输入 Prompt 的结构是否清晰。多个信息来源需要合理拼接。

**最终 Prompt 的结构**（按顺序）：

```
1. [系统角色指令] — 你是谁，应该怎么回答问题
   （医疗助手、安全约束、回答风格参考偏好）

2. [短期记忆] — 最近 5 轮对话 + 会话状态
   （让 AI 知道刚刚发生了什么）

3. [长期记忆] — 患者画像 + 关键事件 + 偏好
   （让 AI 了解这位患者的长期情况）

4. [知识上下文] — RAG 检索到的医学知识块
   （提供外部知识支撑）

5. [工具执行结果] — 刚才调用的工具返回的数据
   （最新的原始数据）

6. [用户问题] — 本轮用户输入
```

## 第三部分：架构设计模式总结

### 模式 1：多层缓存 / 记忆金字塔

```
                        查询频率高 / 容量小
    ↑ 工作记忆（内存中，最近5轮）
    │ 会话缓冲（匿名，瞬态）
    │ 短期消息（DB，保留有限时间）
    │ 关键事件（DB，长期，去重合并）
    │ 画像（DB，持续更新摘要）
    ↓ 偏好（DB，几乎不变）
                        查询频率低 / 容量大
```

越往上，越热、越小、越临时；越往下，越冷、越大、越持久。

### 模式 2：规则优先 + LLM 兜底

在意图识别、实体提取、身份解析等多个环节，都采用：

```
先尝试确定性规则（快、可解释、零成本）
   → 失败时调用 LLM（慢、不可解释、有成本）
```

这避免了每个请求都花 LLM 开销，同时保证了复杂场景的覆盖。

### 模式 3：降级链（Graceful Degradation）

在嵌入层、视觉层、语音层都实现了降级链：

```
首选方案 → 次选方案 → 本地兜底方案
```

确保系统依赖的外部服务不可用时不会整体崩溃。

### 模式 4：数据流水线（Pipeline）

```
原始输入 → 身份解析 → 记忆构建 → 上下文组装 → 工具执行 → 答案生成
```

每个阶段只依赖前一阶段的输出，可插拔、可独立测试、可单独缓存。

### 模式 5：匿名 → 实名过渡

```
匿名会话缓冲（不关联患者 ID）
    → promote（身份确认后迁移到正式表）
    → 正式短期消息（关联患者 ID）
```

这个模式解决了"不知道是谁时的对话数据怎么办"的问题——不丢失数据，也不污染数据。

## 第四部分：涉及的独立知识点清单

1. **HMAC-SHA256 签名**：身份 token 防篡改的基础
2. **Base64 URL-Safe 编码**：token 载荷的序列化方式
3. **余弦相似度**：向量检索的相似度度量
4. **混合检索加权**：多信号融合的排序策略
5. **N-gram 分词**：中文文本的关键词提取（2-gram / 3-gram）
6. **一致性哈希嵌入**：HashTextEmbedder 的确定性文本转向量算法
7. **共识投票**：多候选计划的选择策略
8. **Pydantic 模型校验**：请求/响应的声明式 schema 定义
