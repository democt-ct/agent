## 1. 项目一句话概述

这是一个**患者侧智能辅助 Agent 原型系统**，核心能力是在医疗场景下，让 AI 能"记住患者是谁、说过什么、问过什么"，并基于四层记忆架构（事实 + 工作 + 长期摘要 + 知识）给出可解释、可追溯的回答。

## 3. 四层记忆架构（重点）

这是整个项目最核心的设计，面试中一定会被问到。

### 3.1 事实层 (Factual Memory)

**存什么：** 患者主档（姓名、住址、过敏史、紧急联系人）、病历、就诊记录、关键事件。

**特点：**

- 最底层、最可信的数据来源
- 从 SQLAlchemy ORM 直接读取结构化数据
- 回答"我的住址是什么"这类问题，必须命中事实层，不能靠模型猜
- 当前实现：请求期从 DB 拼装，尚未做持久化向量化

**关键表：** `patients`, `medical_records`, `visit_records`

### 3.2 短期工作记忆 (Working Memory)

**存什么：** 最近几轮对话的结构化摘要，包含：

| 字段                | 作用                     |
| ------------------- | ------------------------ |
| `recent_messages` | 最近 N 条对话原文        |
| `intent`          | 当前对话意图             |
| `current_topic`   | 当前主题                 |
| `goal`            | 当前目标                 |
| `working_summary` | 本轮对话核心概括         |
| `next_action`     | 下一步推理建议           |
| `memory_focus`    | 当前记忆关注范围         |
| `active_entities` | 活跃实体（如疾病、症状） |
| `risk_signals`    | 风险信号                 |

**特点：**

- 随每一轮对话自动更新
- 不是简单的消息堆叠，而是**结构化工作记忆**
- 服务"当前任务推进"，多轮追问时确保上下文连续

**核心设计原则：** 好的工作记忆应服务"当前任务推进"，核心字段是 intent/topic/goal/next_action。

### 3.3 长期摘要记忆 (Long-term Summary Memory)

**存什么：**

| 对象                          | 内容         |
| ----------------------------- | ------------ |
| `MemoryUserProfile`         | 用户画像摘要 |
| `MemoryBusinessProfile`     | 业务画像摘要 |
| `MemoryConversationProfile` | 会话画像摘要 |
| `MemoryPreference`          | 偏好设置     |
| `MemoryKeyEvent`            | 关键事件     |

**特点：**

- 职责是沉淀**稳定画像、偏好、长期关注点**
- 不是原始事实数据库，是抽象后的摘要
- 需注意去重、更新、失效策略
- 不能把所有历史信息都塞进 profile_summary

### 3.4 知识记忆 (Knowledge Memory)

**存什么：** 可复用的医疗知识块（用药指导、检查说明、疾病科普等）。

**存储方式：**

- SQL 元数据表 `memory_knowledge_chunks`（标题、正文、标签、域、来源等）
- ChromaDB 向量索引（持久化到 `data/chroma_knowledge/`）

**检索方式：** 混合检索（见下一节）

## 4. 混合检索 RAG（核心亮点）

### 4.1 为什么要混合检索？

单一向量检索的盲区：精确术语匹配不佳（如药名、疾病名）。混合检索用多路召回来弥补。

### 4.2 三路召回 + 四维打分

**召回通路：**

| 通路       | 实现                           | 代码位置                      |
| ---------- | ------------------------------ | ----------------------------- |
| 向量召回   | ChromaDB 语义相似度搜索        | `_query_chroma_ids()`       |
| 关键词召回 | SQL 查询 + 中文分词 token 匹配 | `_fetch_candidate_chunks()` |
| 元数据过滤 | hospital_id / domain 范围约束  | 两路中均内置                  |

**综合排序权重（`search()` 方法）：**

| 维度                | 权重 | 方法                                   |
| ------------------- | ---- | -------------------------------------- |
| 向量分 (vector)     | 58%  | 余弦相似度 (`_cosine_similarity`)    |
| 关键词分 (keyword)  | 22%  | Token 命中计数 (`_score_keywords`)   |
| 元数据分 (metadata) | 12%  | hospital 匹配 + domain 匹配 + 时间窗口 |
| 时效分 (recency)    | 8%   | 基于 updated_at 的衰减函数             |
| 置信度              | 4%   | chunk.confidence 加权                  |

**最终分数公式：**

```
final_score = 0.58 × vector_score + 0.22 × keyword_score
            + 0.12 × metadata_score + 0.08 × recency_score
            + min(confidence, 1.0) × 0.04
```

**过滤条件：**

- `final_score <= 0.01` 的结果直接丢弃
- `_allowed_for_time_window()` 过滤未生效/已过期的知识块
- 非所属 hospital 的 chunk 只有全局 chunk (hospital_id=null) 才给部分分数

### 4.3 中文分词策略

`_split_tokens()` - **零依赖的轻量中文分词**：

```
中文字符 -> 重叠双字 n-gram + 三字 n-gram
英文/数字 -> 保持完整 token
```

示例："高血压用药" -> `["高血压", "血压用", "压用药", "高血", "血压", "压用", "药用", "用药"]`

**为什么不用 jieba？** 减少依赖，在短查询和标题场景下 n-gram 足够。

### 4.4 关键词打分细节（`_score_keywords()`）

- 匹配在**标题**中：+2 分
- 匹配在正文/tags/domain/source_type 中：+1 分
- 最终分数 = hits / max(2.0, token_count)，归一化到 [0, 1]

### 4.5 元数据打分细节（`_score_metadata()`）

| 条件                            | 加分  |
| ------------------------------- | ----- |
| hospital_id 精确匹配            | +0.55 |
| chunk 为全局 (hospital_id=null) | +0.25 |
| domain 精确匹配                 | +0.25 |
| 在有效时间窗口内                | +0.10 |

## 5. Embedding 模型与 GPU 加速

### 5.1 模型选择

**模型：** `BAAI/bge-small-zh-v1.5`

- 智源研究院 (BAAI) 开源的中文 embedding 模型
- `small` = 轻量版，适合本地部署
- `zh` = 中文场景优化
- `v1.5` = 迭代优化版本

### 5.2 后端选择逻辑（`_build_embedding_backend()`）

```
1. 如果配置了 EMBEDDING_API_BASE/KEY → OpenAI 兼容 embedding
2. 否则 → sentence-transformers (BGE 模型)
3. 都不可用 → HashTextEmbedder（纯本地哈希降级方案）
```

### 5.3 GPU 加速（`_resolve_hf_device()`）

```python

```

模型通过 `HuggingFaceEmbeddings(model_kwargs={"device": "cuda"})` 加载到 GPU。

### 5.4 启动预热（`warmup()`）

在 `app/main.py` 的 `on_startup` 中调用 `get_knowledge_retriever().warmup()`：

- 提前加载 embedding 模型到显存
- 首次同步 ChromaDB 索引
- 避免首个用户请求的冷启动延迟（首次推理可能需 5-10 秒）

### 5.5 降级方案：HashTextEmbedder

当 sentence-transformers 和 OpenAI embeddings 都不可用时：

- 将文本 token 哈希到 256 维向量
- 纯 CPU 计算，零依赖
- 准确度低于 BGE，但保证系统可用

## 7. MCP Server 做了什么？

**MCP (Model Context Protocol)** -- 自研实现，模块化工具调用。

**核心工具：**

- `issue_identity_token` - 发放身份 token
- `verify_identity` - 验证身份
- `get_medical_records` - 获取病历
- `get_visit_records` - 获取就诊记录
- `get_patient_profile` - 获取患者聚合 profile

**设计思路：** 不是用 LangChain Agent，而是**自研 Agent 路由 + LangChain embeddings 封装**。这样可以更精细地控制工具调用逻辑和记忆注入流程。

## 9. 面试常见追问

### 架构类

**Q1: 为什么是四层记忆，而不是一个统一记忆层？**

答案：因为不同信息本质不同，必须分层治理：

| 层       | 性质                       | 存取要求             |
| -------- | -------------------------- | -------------------- |
| 事实层   | 原始结构化数据，精确不可变 | 精确查询、可回溯     |
| 工作记忆 | 当前会话状态，高频变化     | 低延迟、自动更新     |
| 长期摘要 | 稳定画像/偏好，低频变化    | 去重、合并、失效管理 |
| 知识记忆 | 通用医疗知识，可复用       | 语义检索、版本管理   |

如果混在一起，会出现三类严重问题：

1. 事实被摘要污染——患者的真实过敏史被模型"概括"后失真
2. 会话噪声进入长期记忆——闲聊内容变成永久画像
3. 知识检索命中患者隐私——通用知识块检索却返回了其他患者的数据

**Q2: 事实记忆和知识记忆的区别是什么？**

答案：

| 维度       | 事实记忆                         | 知识记忆                     |
| ---------- | -------------------------------- | ---------------------------- |
| 数据归属   | 某个特定患者                     | 所有人共享                   |
| 数据格式   | 结构化字段（住址、过敏史、病历） | 非结构化文本块（文章、指南） |
| 检索方式   | SQL 精确查询为主                 | 向量语义 + 关键词混合        |
| 可信度要求 | 必须 100% 准确                   | 可带置信度分数               |
| 例子       | "张三的住址是 XX 路 1 号"        | "高血压患者应低盐饮食"       |
| 数据量     | 小（单患者百条级）               | 大（数万知识块）             |

回答患者问题时，如果问题是"我有没有过敏史"，必须走事实层；如果是"过敏了该吃什么药"，则需要知识层。

### 检索类

**Q3: 混合检索的权重为什么这样设？**

答案：

```
向量 58% + 关键词 22% + 元数据 12% + 时效 8% + 置信度 4%
```

权重设定的三条原则：

1. **语义理解是核心（58%）**——大部分患者问题是自然语言，语义匹配决定了检索结果的基本质量。向量检索能泛化理解同义表达，比如"胸闷"和"胸口发闷"。
2. **关键词是精确保障（22%）**——精确医学术语（药名、病名、检查名称）不能靠语义模糊匹配。比如"阿托伐他汀"和"瑞舒伐他汀"语义接近但药理不同，关键词匹配能区分它们。
3. **元数据和时效做边界约束（12%+8%）**——防止跨医院、跨域、过时知识的错误召回。比如 A 医院的用药指南不应推荐给 B 医院的患者。

当前权重是经验值，后续需要通过评测集（标注好的 question→正确chunk 对）进行网格搜索或学习调优。

**Q4: 如果 ChromaDB 挂了，系统还能用吗？**

答案：能用，两层降级保护：

```
ChromaDB 挂了 → _query_chroma_ids() 返回空列表
               → 自动退化到纯 SQL + 关键词检索

Embedding 模型也挂了 → LangChainTextEmbedder 抛异常
                     → 自动退化到 HashTextEmbedder（纯本地哈希）
```

具体代码逻辑：

- `_query_chroma_ids()` 中 `collection is None` 时直接返回 `[]`，不抛异常
- `LangChainTextEmbedder.embed_query()` 中 `try/except` 后走 `self._fallback`
- `_fetch_candidate_chunks()` 完全不依赖 ChromaDB，纯 SQL 查询

最坏情况下，系统降级为纯 SQL LIKE + 哈希向量的关键词匹配，虽然准确度下降，但不会完全不可用。

**Q5: 向量检索和关键词检索各自负责什么场景？**

答案：

| 场景类型     | 检索方式   | 示例                               |
| ------------ | ---------- | ---------------------------------- |
| 语义泛化问题 | 向量为主   | "我最近总是胸口不舒服，该注意什么" |
| 精确实体查询 | 关键词为主 | "阿托伐他汀钙片的副作用"           |
| 混合场景     | 两者互补   | "糖尿病人在饮食上要注意什么"       |

具体到中文场景，关键词检索更重要，因为：

- 中文医疗术语同义词少，精确匹配即可覆盖大部分需求
- 拼音缩写（如"CT"、"MRI"）在向量空间中容易被稀释
- n-gram 分词（双字+三字）天然适合中文短词匹配

### 模型与性能类

**Q6: 为什么选 BGE-small 而不是其他模型？**

答案：

| 考量     | BGE-small-zh-v1.5     | 替代方案对比                     |
| -------- | --------------------- | -------------------------------- |
| 中文优化 | 专门针对中文训练      | text-embedding-3-small 偏英文    |
| 模型体积 | ~100MB，小            | bge-large-zh 约 1.3GB            |
| 推理速度 | CPU 上 10-50ms/条     | 大模型需 GPU 或 >200ms           |
| 部署方式 | 本地部署，无 API 费用 | OpenAI embeddings 需网络+付费    |
| 评测成绩 | C-MTEB 中文榜单前列   | —                               |
| 可替换性 | 环境变量一键切换      | `KNOWLEDGE_HF_EMBEDDING_MODEL` |

选择逻辑：原型阶段需要本地可跑、推理快、零费用，BGE-small 是最优解。生产环境可以平滑升级到 bge-large 或 BGE-M3（多语言），只需改环境变量。

**Q7: 预热做了什么？为什么要预热？**

答案：

预热在 `app/main.py` 的 `on_startup` 事件中触发：

```python
@app.on_event("startup")
def on_startup():
    Base.metadata.create_all(bind=engine)
    get_knowledge_retriever().warmup()
```

`warmup()` 方法做了两件事：

1. 调 `embedder.warmup()` → 触发 `HuggingFaceEmbeddings` 初始化，下载/加载 BGE 模型到内存/显存
2. 调 `embedder.embed_query("患者复诊与病历摘要")` → 触发首次推理，确保模型已就绪

为什么要预热：

- 首次加载 sentence-transformers 模型需要 3-10 秒（下载 tokenizer、加载权重）
- 如果没有预热，第一个真实用户请求会卡顿 3-10 秒
- 预热放在启动时，启动慢几秒可接受，但用户的首次对话体验不受影响

**Q8: embedding 缓存怎么做的？**

答案：

```python
_chunk_embedding_cache: Dict[str, tuple[str, List[float]]]
# key = chunk_id
# value = (cache_key, embedding_vector)

def _chunk_embedding_cache_key(chunk):
    # cache_key = "chunk_id:updated_at_iso"
    # chunk 更新后 cache_key 变化 → 缓存自动失效
```

缓存策略：

- 以 chunk_id 为 key，存储 (缓存键, 嵌入向量) 元组
- 缓存键 = `chunk_id:updated_at.isoformat()`，chunk 内容变化时 updated_at 变 → 缓存自动失效
- 加 `threading.Lock` 保证线程安全
- 属于进程内缓存，进程重启后清空，需要重新计算

为什么不用 Redis/外部缓存：当前单进程原型，进程内缓存足够。生产环境可替换为 Redis 缓存，多个 worker 共享。

**Q10: 为什么当前不是 LangChain Agent？**

答案：

LangChain Agent 的问题：

1. **黑盒抽象**——工具调用流程被封装，出问题时很难定位是 prompt 问题、工具选择问题还是记忆注入问题
2. **记忆系统僵硬**——LangChain 的 ConversationBufferMemory 等是通用方案，不适合四层记忆的精细控制
3. **中文优化不足**——LangChain 默认 prompt 和 chain 设计偏英文场景

当前方案：自研 Agent 路由 + 仅复用 LangChain 的 embeddings 封装层

- 工具调用流程自己写 → 可精确控制每一步
- 记忆注入流程自己写 → 可以按四层记忆分别拼装
- 只借用 LangChain 的 `HuggingFaceEmbeddings` → 省去自己封装 sentence-transformers 的工作

### 设计判断类

**Q11: 什么时候 knowledge_memory 为空是正常的？**

答案：以下三种情况 knowledge_memory 为空完全正常：

1. **知识库尚未入库相关内容**——比如患者问具体的个人病史，这应该走事实层，不是知识层
2. **query 太具体或太生僻**——比如问某个罕见病的某个新药，知识库覆盖不到
3. **属于纯患者事实问题**——"我的住址是什么"、"我的主治医生是谁"，命中事实层就够了

判断逻辑：knowledge_memory 为空时，不要强行返回不相关或低分知识块。系统此时应：

- 优先依赖 factual_memory（患者事实）
- 其次依赖 long_term_summary_memory（历史对话摘要）
- 最后用模型自身知识兜底（但需要加"请以医生诊断为准"的免责声明）

**Q12: 医疗场景为什么"事实优先于知识块，知识块优先于自由生成"？**

答案：

这是医疗 AI 的核心安全原则，按可信度从高到低排序：

| 优先级  | 数据来源                   | 可信度 | 为什么                             |
| ------- | -------------------------- | ------ | ---------------------------------- |
| 第1优先 | 患者事实（病历、主档）     | 最高   | 结构化数据，可追溯，有记录可查     |
| 第2优先 | 知识块（审核过的医疗知识） | 较高   | 经过人工审核或权威来源，带置信度分 |
| 第3优先 | 长期摘要（画像/偏好）      | 中等   | 从历史对话中抽取，有失真风险       |
| 第4优先 | 模型自由生成               | 最低   | LLM 可能幻觉，可能编造不存在的信息 |

实际例子：

- 患者问"我对什么药过敏？"→ 必须从事实际查询，模型不能猜
- 患者问"这个药有什么副作用？"→ 优先查知识块，知识块没有再用模型知识，但需加免责声明
- 患者问"我应该怎么办？"→ 不能直接给诊断建议，应回复"建议您到医院进一步检查"

这就是为什么代码中 `final_score` 低于 0.01 的 chunk 直接丢弃——宁可少召回，不能召回错的。
