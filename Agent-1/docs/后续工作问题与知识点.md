## 1. 你接下来最值得推进的问题

### 1.1 工作记忆质量问题

你需要回答：

- 当前 `working_summary` 是否真的能概括本轮对话核心？
- `next_action` 是否对下一轮推理有帮助？
- `memory_focus` 的规则是否足够稳定？
- `recent_messages` 的保留长度是否合理？
- 多轮追问时，旧主题和新主题如何切换？

你要记住：

- 短期记忆不等于消息堆叠
- 好的工作记忆应该服务“当前任务推进”
- 工作记忆核心字段是：`intent / topic / goal / next_action（意图/主题/目标/下一步行动）`

### 1.2 长期摘要记忆质量问题

你需要回答：

- `MemoryUserProfile`、`MemoryBusinessProfile`、`MemoryConversationProfile` 的职责边界是否清晰？
- 哪些信息应该进入长期摘要，哪些不应该？
- 关键事件如何去重、更新、失效？
- 偏好信息是否会被错误覆盖？

你要记住：

- 长期摘要记忆不是原始事实数据库
- 它的职责是“沉淀稳定画像、偏好、长期关注点”
- 不能把所有历史信息都塞进 profile_summary

### 1.3 患者事实记忆问题

你需要回答：

- 当前事实记忆为什么是“请求期临时拼装”？
- 是否需要把患者主档、病历、就诊、关键事件切成持久化 chunk？
- 事实记忆和长期摘要记忆如何避免重复？
- 回答“住址、过敏史、上次就诊、复诊建议”时，优先读哪层？

你要记住：

- 事实记忆的核心价值是“可引用、可回溯、可信”
- 医疗场景优先事实，不要优先生成
- 患者主档、病历、就诊记录应当视为事实源，不是摘要源

### 1.4 混合 RAG 质量问题

你需要回答：

- 当前向量召回和关键词召回各自承担什么职责？
- 当前排序分数权重是否合理？
- 哪些问题更依赖关键词精确命中，哪些更依赖语义理解？
- 什么时候 `knowledge_memory` 为空是正常的？
- 什么时候 `factual_memory` 应该优先于 `knowledge_memory`？

你要记住：

- 向量召回负责语义
- 关键词召回负责精确命中
- 元数据过滤负责约束范围
- rerank 负责最终排序
- 医疗场景里“事实优先于知识块，知识块优先于自由生成”

### 1.5 Memory Debug 观测问题

你需要回答：

- 当前调试面板是否足够帮助定位问题？
- 中文卡片是不是已经覆盖日常测试诉求？
- 哪些场景还必须看原始 JSON？
- 是否需要增加命中来源标识，比如“来自患者主档 / 病历 / 知识块”？

你要记住：

- 调试面板的职责不是“展示所有数据”
- 而是“帮助快速判断问题在哪一层”

## 2. 你必须记住的核心知识点

### 2.1 四层记忆架构

你必须非常清楚这四层：

1. `事实层`

- 患者主档
- 病历
- 就诊记录
- 关键事件

2. `短期工作记忆`

- recent messages
- intent
- topic
- goal
- next action

3. `长期摘要记忆`

- user profile
- business profile
- conversation profile
- preference

4. `知识记忆`

- 知识块
- ChromaDB
- 混合检索

### 2.2 为什么不能把所有东西都叫长期记忆

因为三类东西本质不同：

- 原始事实：应该精确、可回溯
- 摘要画像：应该稳定、抽象、可复用
- 会话记忆：应该服务当前任务推进

这三类如果混掉，后面一定会越来越乱。

### 2.3 为什么 Memory Debug 很重要

因为你现在不是在“做一个黑盒聊天机器人”，而是在做：

- 可解释的 Agent
- 可验证的记忆系统
- 可分层调试的问答链路

所以你每次测试，都要能回答：

- 这轮命中了哪层记忆？
- 没命中是为什么？
- 是路由问题、抽取问题、检索问题还是展示问题？

### 2.4 当前 embedding 与检索知识点

你必须记住：

- 当前 embedding 模型：`BAAI/bge-small-zh-v1.5`
- 当前 embedding 后端：`sentence-transformers`
- 当前向量库：`ChromaDB`
- 当前检索方式：`混合检索`
- 当前不是纯 LangChain Agent
- 当前是“自研 Agent 路由 + LangChain embeddings 封装”

### 2.5 当前工程边界

你必须清楚哪些已经做了，哪些没做：

已经做了：

- 数据层
- MCP 层
- 记忆聊天
- 混合 RAG
- Memory Debug

还没正式做完：

- React 正式前端
- 定时刷新与后台调度
- 更强 rerank
- 生产级权限与审计

## 3. 你后续工作时最该自问的问题

下面这些问题，建议你反复看。

### 架构问题

- 当前这项能力属于事实层、工作记忆层、长期摘要层，还是知识层？
- 它应该被持久化吗？
- 它应该被向量化吗？
- 它应该参与每轮回答吗？

### 记忆问题

- 这条信息是“稳定事实”还是“当前会话状态”？
- 这条信息应该进 profile、key event，还是只留在 recent messages？
- 用户下次再问时，系统应该优先从哪层找到它？

### 检索问题

- 这个问题更适合事实检索还是知识检索？
- 召回不到是因为没有数据，还是因为排序不对？
- 当前命中的证据片段是否足够支持回答？

### 产品问题

- 用户真正想知道的是事实、解释，还是建议动作？
- 这轮回答需要“给答案”还是“先确认身份/范围”？
- 高风险场景下应该直接回答，还是加限制和提示？

### 调试问题

- 当前错误出在前端展示、接口返回、上下文拼接、还是模型回答？
- `Memory Debug` 是否已经提供了足够证据？
- 如果还不够，需要加哪种可观测信息？

## 4. 当前最建议推进的三件事

### 4.1 把患者事实记忆正式持久化

目标：

- 不再只在请求期临时拼装
- 把主档 / 病历 / 就诊 / 关键事件切成标准化 chunk
- 进入统一检索层

### 4.2 建立问题集和验收集

目标：

- 不靠感觉测
- 要有一批稳定问题
- 每次改记忆或检索，都能回归验证

### 4.3 做更强的排序与评估

目标：

- 明确哪些问题现在答得不稳
- 用评测数据看优化是否真的生效

## 5. 当前结论

你现在最重要的不是继续加更多名词，而是把这套系统真正吃透：

- 哪一层存什么
- 哪一层负责什么
- 哪一层出了问题怎么查

如果你能把这份文档里的问题都答清楚，你对当前项目的理解就已经过线了。
