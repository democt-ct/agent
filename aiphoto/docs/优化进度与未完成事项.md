# AI Photo Agent 优化进度与未完成事项

> 本文档记录基于 `docs/` 设计文档对项目的全面优化进度。每个阶段独立可交付，已完成的已通过编译/导入验证。

## ✅ 已完成阶段

### 阶段 1：对话迭代修复（核心）

**问题**：`chat.py` 每次都基于 `request.image_id`（原始上传图）重新处理，导致"再亮一点"丢失前次结果——迭代编辑被破坏。

**改动**：
- `backend/app/api/endpoints/chat.py`：`ChatRequest` 增加 `current_image_id: Optional[str]`；优先用它作为编辑起点，新上传图自动重置链
- `frontend/src/pages/index.tsx`：维护 `currentImageId` 状态，每次 AI 回复后更新；维护 `originalMap` 追踪每张图的原始上传 id
- 后端响应新增 `params` / `diagnosis` / `applied_params` / `source_image_id` 字段，供前端可视化

**验证**：✅ 前端 `tsc --noEmit` 通过，后端 Python 导入正常

---

### 阶段 2：激活精细控制组件 + 对比滑块

**问题**：`ImageEditor.tsx` + `ImageUploader.tsx`（12 滑块 / 13 预设 / 分析面板）已成孤儿组件，主界面只用硬编码聊天 UI；`react-image-comparison-slider` 已装但未用。

**改动**：

- 对话模式 AI 结果图下方加 `↔ 对比原图` 按钮（弹出 CompareSlider）和 `🎨 专业微调` 按钮（跳转专业模式并加载该图）
- 专业模式用 `dynamic()` 懒加载 `ImageEditor`，避免 SSR 问题

**验证**：✅ 前端类型检查通过

### 阶段 3：Memory / 历史记录

**问题**：SAD 文档承诺 `users / edit_history / user_preferences` 表，实际无持久化，刷新即丢。

**改动**：
- 新建 `backend/app/services/history_service.py`：基于 `backend/data/` 的 JSON 文件存储
  - `sessions.json`：会话列表
  - `messages_{sid}.json`：每会话消息流
  - `preferences.json`：长期偏好（风格偏好计数、平均参数偏移，增量均值）
- 新建 `backend/app/api/endpoints/history.py`：
  - `GET/POST /api/v1/history/sessions` — 列出 / 创建会话
  - `GET /api/v1/history/sessions/{id}` — 取会话
  - `GET/POST /api/v1/history/sessions/{id}/messages` — 取 / 追加消息
  - `GET /api/v1/history/preferences` — 取用户偏好
- `chat.py`：接受 `session_id`，自动追加消息 + `record_edit()` 累积偏好；`_preference_hint()` 注入用户风格偏好到 prompt
- `index.tsx`：启动自动创建会话；Header 加 `＋ 新会话` / `🕘 历史` 按钮；右侧抽屉显示会话列表，点击恢复

**验证**：✅ 后端导入通过，前端类型检查通过

**已知限制**：JSON 文件存储非线程安全，并发下可能丢数据；MVP 单用户场景足够，预留接口便于切 PG/Redis。

## ⏳ 未完成阶段

### 阶段 4a：Reflection 闭环（Evaluator 自评打分重试）

**目标**：兑现 SAD 文档 "Evaluator Agent + Reflection Workflow" 的设计——编辑后自动评估质量，分数低则重试。

**计划改动**：
- `backend/app/services/vision_service.py`：新增 `evaluate(original_path, edited_path, user_goal) -> {score, reason}` 方法
  - Prompt：输入原图 + 结果 + 用户目标，输出 `{"score": 0-100, "reason": "..."}`
  - 复用现有 `chat()` 的 base64 + Qwen-VL 调用
- `backend/app/api/endpoints/chat.py`：在 `apply_edits()` 之后调用 evaluate
  - 若 `score < 75` 且重试次数 < 2：把上次参数 + 评语注入 prompt，重新生成 → 再编辑 → 再评估
  - 响应新增 `score` / `retry_count` 字段
- `ChatResponse` 增加字段：`score: Optional[int]`、`retry_count: Optional[int]`
- `frontend/src/pages/index.tsx`：AI 结果消息上显示 `⚡ 自评 85分` 徽标；低分显示 `⚠️ 已重试`

**预估工作量**：中等（核心逻辑约 80 行 Python + 30 行 TSX）

**风险**：每次编辑增加 1 次 LLM 调用，响应 +2~3s；建议与 4b 流式输出一起做，用进度提示掩盖延迟

---

### 阶段 4b：SSE 流式输出

**目标**：避免干等 5-10s，分阶段推送进度。

**计划改动**：
- `backend/app/api/endpoints/chat.py`：新增 `POST /api/v1/chat/stream`
  - 用 `fastapi.responses.StreamingResponse` + `media_type="text/event-stream"`
  - 分阶段 yield SSE 事件：
    ```
    event: stage    data: {"stage":"thinking"}
    event: stage    data: {"stage":"editing","params":{...}}
    event: stage    data: {"stage":"reflecting"}      ← 依赖 4a
    event: done     data: {"image_id":"...","reply":"...","score":85}
    ```
- `frontend/src/pages/index.tsx`：用 `EventSource` 替换主编辑流程的 axios 调用
  - loading 气泡文字动态切换：`思考中…` → `调色中…` → `评估中…`
- 纯文本对话（无图）保持现有 axios 调用不变

**预估工作量**：中等（后端约 100 行，前端约 50 行）

**风险**：SSE 在某些代理下需配置 `X-Accel-Buffering: no`；前端 EventSource 不支持 POST，需用 fetch + ReadableStream 或 POST-to-GET 变通

### 阶段 4c：参数可视化卡片 + 诊断条形图

**目标**：把 AI 输出从"正则解析文本"升级为"结构化可视化卡片"。

**计划改动**：

  - 每条带数值标签 + 方向箭头
- 新建 `frontend/src/components/DiagnosisBars.tsx`：
  - 接收 `diagnosis: DiagData`
  - 3 条 mini bar：亮度 / 对比度 / 饱和度（0-255 标准化），标注"偏暗/正常/偏亮"
- `index.tsx`：`renderAIReply()` 重构——优先用 `m.appliedParams` 渲染 `<ParamCard>`，文本正则解析作为 fallback
  - 同时渲染 `<DiagnosisBars diagnosis={m.diagnosis} />`

**预估工作量**：小（两个纯展示组件，约 150 行）

**依赖**：阶段 1 已铺好 `applied_params` / `diagnosis` 字段，此阶段纯前端

### 阶段 4d：多图会话

**目标**：一次会话内切换多张图独立编辑，各图保留各自的编辑链。

**计划改动**：
- `frontend/src/pages/index.tsx`：
  - 数据结构升级：`images: Record<string, { originalId: string; currentId: string; }>` 替代单一 `currentImageId`
  - 顶部横向缩略图条，点击切换"当前编辑的图"
  - `send()` 根据 `activeImageId` 选择对应的 `currentId` 作为链式起点
- 后端无需改动（`current_image_id` 已支持任意图 id）

**预估工作量**：中等（前端状态管理重构，约 100 行）

**风险**：与阶段 3 会话恢复逻辑交互复杂——恢复历史会话时需重建 `images` 映射；建议放在最后做

## 🔧 技术债与可选优化（非阶段计划）

以下是实施过程中发现、但不在原计划内的改进点，可按需处理：

### 后端
- `backend/app/models/database.py`：SQLAlchemy 模型定义存在但未实际使用（项目用 JSON 文件）。可删除或迁移 history_service 到 SQLAlchemy
- `backend/app/services/vision_service.py:9`：API URL 硬编码，应移入 `config.py`
- `image_service.py:29`：`get_image_path` 只查 4 种扩展名，HEIC/AVIF 等新格式不支持
- `editing_service.py:313`：`compare_images` 返回硬编码假数据 `{"brightness_change": 0.2}`，应真实计算
- CORS `allow_origins=["*"]` 生产环境需收紧

### 前端
- `_app.tsx` 未引入 Tailwind（虽然 globals.css 引了），`ImageEditor.tsx` 用 Tailwind class，需确认构建时 CSS 生效
- `next.config.js` 代理到 `:8011`，但 `README/SETUP` 写的是 `:8000`，端口不一致
- 所有内联 style 应抽取为 CSS module / styled-components，提升可维护性
- 缺少 loading skeleton 和错误重试 UI

### 架构
- 当前是"伪多 Agent"——实际只有一个 `/chat` 端点做 LLM 调用。若要真正兑现 SAD 的 Multi-Agent 设计，需引入 LangGraph 或自建状态机编排 Vision/Critic/Planner/Editing/Evaluator 节点
