# 可填写并落库的聊天模板说明

本文档记录当前聊天界面中已经可以通过表单填写，并能真实写入后端数据的模板能力，以及后续新增类似功能时需要关注的改动点。

## 当前已支持的落库模板

| 模板 | 聊天卡片类型 | 触发示例 | 提交接口 | 数据/记录 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 请假申请 | `leave_form` | `我要请假`、`申请年假`、`帮我请假` | `POST /api/leaves` | 请假记录 | 会进入申请记录和审批流程。 |
| 报销申请 | `expense_form` | `我要报销`、`申请报销`、`提交报销` | `POST /api/expenses` | 报销记录 | 会进入报销记录/申请记录。 |
| IT 维修工单 | `ticket_form` | `修电脑`、`电脑坏了`、`电脑开不了机，帮我报修` | `POST /api/tickets` | `it_tickets` 工单记录 | 会出现在“我的申请 → IT 工单”中。 |
| 考勤打卡 | `attendance_punch` | `打卡`、`上班打卡`、`下班打卡` | `POST /api/attendance/punch` | 考勤打卡记录 | 这是即时操作，不是申请单，但会写入后端记录。 |

## 相关代码位置

### 1. 聊天意图识别

文件：`src/agents/orchestrator.py`

核心位置：

- `ACTION_CARD_RULES`
- `_detect_action_card()`

作用：根据用户输入识别是否需要返回聊天表单卡片。

例如：

- 用户说“我要请假” → 返回 `leave_form`
- 用户说“我要报销” → 返回 `expense_form`
- 用户说“电脑开不了机，帮我报修” → 返回 `ticket_form`

### 2. 前端聊天卡片渲染和提交

文件：`frontend/app.js`

核心函数：

- `renderActionCard(card)`：根据 `card.type` 渲染对应表单。
- `bindCardEvents(el)`：绑定提交按钮事件，并调用对应后端接口。

当前已落库模板对应关系：

| `card.type` | 前端提交动作 | 后端接口 |
| --- | --- | --- |
| `leave_form` | `submit_leave` | `/api/leaves` |
| `expense_form` | `submit_expense` | `/api/expenses` |
| `ticket_form` | `submit_ticket` | `/api/tickets` |
| `attendance_punch` | `punch_in` / `punch_out` | `/api/attendance/punch` |

### 3. 后端 API 路由

相关文件：

- `src/api/routes/leaves.py`：请假申请接口。
- `src/api/routes/expenses.py`：报销申请接口。
- `src/api/routes/tickets.py`：IT 工单提交接口。
- `src/api/routes/attendance.py`：考勤打卡接口。
- `src/api/routes/applications.py`：统一申请记录查询，当前支持请假、报销、IT 工单。
- `src/api/app.py`：注册 API 路由。

### 4. 业务工具和数据库

相关文件：

- `src/tools/hr_tools.py`：HR/请假相关工具。
- `src/tools/finance_tools.py`：财务/报销相关工具。
- `src/tools/it_tools.py`：IT 工单相关工具。
- `src/tools/db.py`：数据库表结构和 CRUD 方法。

## 后续新增一个可填写并落库模板麻不麻烦？

整体不算麻烦，因为现在已经有一套固定模式可以复用。

如果只是新增一个类似“IT 工单”这种简单表单，通常需要改 4 类地方：

1. **数据库/业务工具**
   - 如果已有数据表和工具函数，就直接复用。
   - 如果没有，需要在 `src/tools/db.py` 增加表和 CRUD 方法。
   - 再在对应领域工具文件里封装业务函数，例如 `submit_xxx()`。

2. **后端 API**
   - 在 `src/api/routes/` 下新增或扩展路由文件。
   - 提供类似 `POST /api/xxx` 的提交接口。
   - 在 `src/api/app.py` 注册路由。

3. **聊天意图规则**
   - 在 `src/agents/orchestrator.py` 的 `ACTION_CARD_RULES` 中加入触发关键词。
   - 指定新的 `card_type`，例如 `asset_request_form`、`trip_form`、`overtime_form` 等。

4. **前端卡片**
   - 在 `frontend/app.js` 的 `renderActionCard()` 中新增卡片表单。
   - 在 `bindCardEvents()` 中新增提交动作，调用对应 API。
   - 提交成功后展示成功提示，并按需刷新列表。

## 推荐的新增模板流程

以后每新增一个“可填写并落库”的模板，建议按这个顺序做：

1. 先确定业务字段
   - 例如：类型、日期、金额、原因、优先级、附件等。

2. 确定是否需要审批
   - 不需要审批：提交后直接生成记录。
   - 需要审批：还要接入审批流和状态流转。

3. 确定列表中是否要展示
   - 如果要出现在“我的申请”里，需要同步扩展 `applications.py` 和相关查询逻辑。

4. 增加后端数据和接口
   - 先保证接口可以直接提交并查到数据。

5. 增加聊天卡片
   - 前端只负责展示表单、提交数据和展示结果。

6. 增加意图触发词
   - 让用户用自然语言就能唤起对应模板。

7. 做一次端到端验证
   - 聊天触发 → 填表 → 提交 → 数据落库 → 列表可见。

## 工作量判断

| 新增类型 | 难度 | 原因 |
| --- | --- | --- |
| 已有数据表和工具，只差聊天卡片 | 低 | 类似这次 IT 工单，主要补前端卡片、API 和触发词。 |
| 没有数据表，但不需要审批 | 中 | 需要新增表、CRUD、API、前端卡片和触发词。 |
| 没有数据表，且需要审批流程 | 中高 | 除了表单和接口，还要接入审批、状态流转、通知、申请列表。 |
| 涉及附件、图片、文件上传 | 中高 | 需要额外处理文件存储、访问权限和前端上传交互。 |
| 涉及复杂多步骤流程 | 高 | 需要拆分状态机、审批链路、多页面/多卡片交互。 |

## 建议

后续如果你要继续添加功能，建议先把每个模板按下面格式定义清楚：

```md
模板名称：
所属部门/专家：
触发话术：
需要填写的字段：
是否需要审批：
提交后进入哪个列表：
是否需要通知：
是否需要附件：
```

这样开发时就可以直接套用现有 `action_card + API + 数据表 + applications` 的模式，成本会比较可控。
