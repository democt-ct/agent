## 阶段一：安全修复（1~3天）

> 这三项存在实际安全风险，应在上线前全部完成。

### Task 1 · JWT 加入过期时间

**文件**：`src/gateway/auth.py`

**现状**：Token 永不过期，泄露即永久沦陷。

**目标**：Access Token 8h 过期 + Refresh Token 7d，支持续期。

**改动步骤**：

1. `create_token()` 加 `"exp": now + timedelta(hours=8)`。
2. 新增 `create_refresh_token(user_id)` 签发 7d refresh token，存入 `refresh_tokens` 表（字段：token_hash, user_id, expires_at, revoked）。
3. 新增路由 `POST /api/auth/refresh`：校验 refresh token → 签发新 access token。
4. `verify_token()` 已有 `ExpiredSignatureError` 处理，无需改动。
5. 前端 `app.js`：拦截 401 响应 → 自动调用 `/api/auth/refresh` → 重发原请求。

**验收**：登录后等待 8h+ token 失效，refresh 接口正常续期；refresh token 吊销后 401。

---

### Task 2 · Session 所有者校验

**文件**：`src/api/routes/chat.py`

**现状**：`session_id` 由客户端传入，未校验归属，任意用户可读他人对话历史。

**目标**：加载 session 前验证当前用户是该 session 的 owner。

**改动步骤**：

1. 在 `chat_stream()` 加载历史处（第 142 行）增加校验：

```python
owner = state.session_owners.get(sid)
if owner and owner != (session_ctx.user_id if session_ctx else "anonymous"):
    sid = uuid.uuid4().hex[:12]   # 非法访问：重开新 session
```

2. `chat_endpoint()` 同步接口做相同校验（`_save_session` 调用前）。
3. 匿名用户（`session_ctx` 为 None）的 session 隔离在 `"anonymous"` 命名空间，不与登录用户共享。

**验收**：用户 A 用 B 的 session_id 发请求，返回全新空白 session，无法读取 B 的历史。

---

### Task 3 · 工具层强制绑定 session user_id

**文件**：`src/agents/tool_agent.py`（工具执行入口）

**现状**：工具参数 `user_id` 来自 LLM tool_call，可被提示注入篡改，越权操作他人数据。

**目标**：所有包含 `user_id` 的工具调用，强制覆盖为当前 session 的 user_id（除 admin 角色外）。

**改动步骤**：

1. 在 `ToolAgent.execute()` 执行工具前添加参数净化：

```python
USER_ID_PARAMS = {"user_id", "applicant_id", "employee_id"}

def _sanitize_args(tool_args: dict, session_ctx) -> dict:
    if session_ctx is None or session_ctx.role == "admin":
        return tool_args
    for key in USER_ID_PARAMS:
        if key in tool_args:
            tool_args[key] = session_ctx.user_id
    return tool_args
```

2. 在 `execute()` 调用实际工具函数前调用 `_sanitize_args()`。
3. 管理员操作需显式在 session_ctx 标记，不能通过修改请求参数升权。

**验收**：构造含 `"user_id": "other_user"` 的 tool_call，执行结果只影响自己数据。

## 阶段二：稳定性修复（3~7天）

### Task 4 · Session 内存泄漏 + 并发竞争

**文件**：`src/api/app.py`、`src/api/routes/chat.py`

**现状**：`app.state.sessions` 是无界字典，session 永不清理；多请求并发写入同一 session 无锁保护。

**目标**：Session TTL 24h + 写入加锁 + 可选持久化到 SQLite。

**改动步骤**：

1. 将 `state.sessions` 替换为 `TTLCache`（使用 `cachetools.TTLCache(maxsize=10000, ttl=86400)`）。
2. `_save_session()` 用 `asyncio.Lock`（每 session_id 一把锁，存 `state.session_locks`）包裹写入。
3. 加入 `state.session_count` 监控指标，在 `/api/health` 暴露当前活跃 session 数。
4. （可选）将 session 持久化到 `conversations` 表，支持重启后恢复。

**验收**：压测 1000 并发对话无竞争异常；24h 后 session 自动清理，内存回落。

### Task 5 · 会话历史截断

**文件**：`src/api/routes/chat.py`（第 143 行）、`src/api/service/chat_service.py`

**现状**：全量历史直接传给 LLM，超长会话超出 context window 或产生极高费用。

**目标**：传给 LLM 的历史最多保留最近 N 轮（默认 10 轮 = 20 条消息）。

**改动步骤**：

1. `config.py` 加 `chat_history_window: int = 10`。
2. `chat_stream()` 加载历史后截断：

```python
history = history[-(app_config.chat_history_window * 2):]
```

3. `chat_endpoint()` 对 `req.history` 同样截断。
4. `SessionMemory` 的 `window_size` 与此值保持一致，统一来源于 config。

**验收**：构造 50 轮历史请求，LLM 收到的 messages 数量 ≤ 20。

### Task 6 · 错误信息脱敏

**文件**：`src/api/routes/chat.py`（第 100、103 行）

**现状**：`detail=str(e)` 将异常原文（含路径、SQL）直接返回客户端。

**目标**：生产环境只返回错误码，详情写日志。

**改动步骤**：

1. `config.py` 加 `debug_mode: bool = os.getenv("DEBUG", "false").lower() == "true"`。
2. 错误响应改为：

```python
detail = str(e) if app_config.debug_mode else "请求处理失败，请稍后重试"
```

3. 同时在 `logger.exception()` 记录完整堆栈（已有，无需改动）。

**验收**：生产环境（`DEBUG=false`）触发异常，响应 body 无路径/SQL 信息。

### Task 7 · LLM 熔断器

**文件**：`src/llm_client.py`

**现状**：LLM API 不可用时每次请求重试 N 次，大量请求堆积耗尽线程池。

**目标**：实现 Circuit Breaker，故障时快速失败，恢复后自动半开探测。

**改动步骤**：

1. 引入 `pybreaker` 库（或手写简单计数器）：

```python
# 5次失败 → 断路 60s → 半开探测
breaker = CircuitBreaker(fail_max=5, reset_timeout=60)

@breaker
def call_llm_with_retry(...): ...
```

2. 断路时抛出 `CircuitBreakerError`，路由层捕获后返回 503 + `Retry-After` header。
3. `/api/health` 暴露熔断器状态（closed/open/half-open）。

**验收**：模拟 LLM API 连续超时，第 6 次请求立即返回 503，60s 后自动恢复探测。

### Task 8 · 环境配置分层

**文件**：`src/config.py`、`.env.example`

**现状**：dev 默认值（弱 JWT 密钥、关闭 rerank）会被生产继承，无环境区分。

**目标**：dev/prod 配置显式分离，生产环境缺失必要配置时启动失败。

**改动步骤**：

1. `config.py` 加 `APP_ENV = os.getenv("APP_ENV", "dev")`。
2. 生产环境启动时校验必要配置：

```python
if APP_ENV == "production":
    assert JWT_SECRET != "dev-secret-change-in-production", "生产环境必须设置 JWT_SECRET"
    assert os.getenv("DEEPSEEK_API_KEY"), "生产环境必须设置 DEEPSEEK_API_KEY"
```

3. `.env.example` 补充所有必要的生产配置项及说明注释。

**验收**：`APP_ENV=production` 且未设置 `JWT_SECRET` 时，服务启动报错退出而非静默使用弱密钥。

### Task 9 · 路由关键词优化（否定句处理）

**文件**：`src/agents/orchestrator.py`

**现状**：关键词匹配不识别否定句，"我不需要请假" 会被路由到 HR。

**目标**：否定语境下降低关键词权重，避免误路由。

**改动步骤**：

1. 在关键词匹配前检测否定前缀：

```python
NEGATION_PATTERNS = re.compile(r"(不|无需|不需要|不想|没有|取消).{0,4}$")

def _is_negated(text: str, keyword: str) -> bool:
    idx = text.find(keyword)
    return bool(NEGATION_PATTERNS.search(text[:idx]))
```

2. 被否定的关键词命中不计入分域得分。
3. 对于否定意图，优先路由到 fallback 或根据对话历史推断真实意图。

**验收**："我不需要请假"不路由到 HR；"我需要请假"正常路由到 HR。

---

### Task 10 · 前端 app.js 模块拆分

**文件**：`frontend/app.js`（45KB 单文件）

**现状**：所有逻辑混在一个文件，可维护性差。

**目标**：按职责拆分为独立 ES Module，不引入构建工具。

**建议拆分**：

```
frontend/
├── app.js              # 入口，仅负责初始化和事件绑定
├── modules/
│   ├── api.js          # 所有 fetch / SSE 请求封装
│   ├── chat.js         # 对话渲染逻辑
│   ├── auth.js         # 登录/登出/token 管理
│   ├── forms.js        # 操作卡片（请假/报销/工单）
│   └── session.js      # Session 管理
```

`index.html` 加 `type="module"` 引入 `app.js`，用 `import` 按需加载各模块。

**验收**：功能不变，单个模块文件 < 8KB，新增功能只需修改对应模块。

## 实施记录（2026-06）

### 阶段一：安全修复

**Task 1** — `gateway/auth.py` 的 `create_token()` 加入 `exp` 字段（8h 过期）；`config.py` 新增 `JWT_EXPIRY_HOURS` 环境变量；`auth` 路由的 login/refresh 端点返回 `expires_in`；`schema.py` 新增 `refresh_tokens` 表供后续吊销功能。前端已有 JWT 自动刷新逻辑（过期前 2 分钟自动续期），无需额外改动。

**Task 2** — `chat.py` 的 `chat_stream()` 和 `chat_endpoint()` 均在加载 session 前校验 `state.session_owners` 所有权。非 owner 使用他人 session_id 时自动重开新 session 且无法读取原会话历史。

**Task 3** — `ToolAgent` 新增 `_sanitize_args()` 方法：非 admin 角色强制将 `user_id`/`applicant_id`/`employee_id`/`approver_id` 覆盖为当前 session 用户值。admin 角色保持原参数不变。在 `execute()` 权限校验之前执行，形成双重防护。

### 阶段二：稳定性修复

**Task 4-5-6** — `config.py` 新增三个配置项：`chat_history_window`（默认 10 轮=20 条）、`debug_mode`（默认 false）、`app_env`（默认 dev）。`chat.py` 的 SSE 和 POST 路径均对历史消息做截断；500 错误的 `detail` 字段在非 debug 模式下脱敏为通用提示。

### 阶段三：架构优化

**Task 7** — `llm_client.py` 新增手写 `CircuitBreaker`（线程安全）：连续 5 次可重试异常→断路 60s→半开探测。`call_llm_with_retry` 调用前检查熔断状态，成功调用重置计数。环境变量 `CB_FAIL_MAX`/`CB_RESET_TIMEOUT` 可调。

**Task 8** — `config.py` 新增 `APP_ENV` 环境变量，模块加载时自动校验：`production` 模式下若 `JWT_SECRET` 为默认值或 `LLM_API_KEY` 未设置，立即 `raise RuntimeError` 阻止启动。

**Task 9** — `orchestrator.py` 新增 `_is_negated()` 函数和 `_NEGATION_PATTERNS` 正则。`_keyword_match()` 在统计关键词命中时过滤被否定词（不/无需/不想/没有/取消）前缀修饰的匹配项。

**Task 10** — `frontend/modules/` 目录新建 `api.js`(2.5KB)、`auth.js`(2.2KB)、`shared.js`(1.3KB) 三个 ES Module。`index.html` 改为 `type="module"` 加载，token 管理/JWT 刷新/fetch 封装/图标常量均模块化。`app.js` 通过 `import` 引用模块，移除 IIFE 内重复定义。
