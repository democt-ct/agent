# 接口协议文档 — Agent 通信规范

> 版本：v0.1  
> 本文档定义系统中所有 Agent 之间的通信协议。团队各方按此规范开发，确保模块可以独立运行并正确对接。

---

## 1. 通信模型

### 1.1 架构原则

```
用户 ──→ Orchestrator ──→ Specialist Agent ──→ Orchestrator ──→ 用户
              │                                              ↑
              └── 所有通信都经过 Orchestrator，不走 P2P ──────┘
```

- **所有通信走 Orchestrator 中转**：Specialist Agent 之间不直接对话
- **同步调用**：Orchestrator 调用 Specialist Agent，等待返回，再做下一步决策
- **无状态设计**：Specialist Agent 不维护跨轮状态，上下文由 Orchestrator 传递给 Agent

### 1.2 调用流程

```
1. 用户输入 → Orchestrator
2. Orchestrator 意图识别 → 确定目标 Agent(s)
3. Orchestrator 构造 AgentRequest → 发送给目标 Agent
4. Specialist Agent 处理 → 返回 AgentResponse
5. Orchestrator 判断：
   a. 如果回答完整 → 返回给用户
   b. 如果需要下一个 Agent → 构造新的 AgentRequest（携带已有上下文）
   c. 如果 Agent 返回错误 → 重试或兜底
```

## 2. Agent 消息格式

### 2.1 AgentRequest（Orchestrator → Specialist）

每个 Specialist Agent 收到且只收到这一个数据结构：

```python
@dataclass
class AgentRequest:
    """Orchestrator 发给 Specialist Agent 的请求"""
    
    # === 基础信息 ===
    query: str                         # 用户原始问题（完整）
    agent_name: str                    # 目标 Agent 名称（如 "hr_agent"）
    
    # === 上下文 ===
    conversation_history: list[dict]   # 历史对话 [{"role": "user"/"assistant", "content": "..."}]
    extracted_entities: dict           # Orchestrator 已提取的实体（可选）
    """
    示例：
    {
        "user_id": "EMP001",
        "department": "Engineering",
        "mentioned_dates": ["2025-03-15"]
    }
    """
    
    # === 跨 Agent 交割上下文 ===
    handoff_context: dict | None       # 来自上一个 Agent 的输出（首次调用为 None）
    """
    示例：HR Agent 处理后交割给 IT Agent：
    {
        "from_agent": "hr_agent",
        "resolved_items": ["病假已审批，按 80% 工资计算"],
        "unresolved_items": ["笔记本报修需要 IT 处理"],
        "pending_tasks": [{"tool": "create_ticket", "params": {...}}]
    }
    """
    
    # === 控制参数 ===
    max_tool_calls: int = 5            # 单次 Agent 最大工具调用次数
    temperature: float = 0.3           # 回答温度（事实性问题用低温）
```

### 2.2 AgentResponse（Specialist → Orchestrator）

```python
@dataclass
class AgentResponse:
    """Specialist Agent 返回给 Orchestrator 的结果"""
    
    agent_name: str                    # 当前 Agent 名称
    
    # === 核心输出 ===
    answer: str                        # 给用户的回答正文（markdown 格式）
    confidence: float                   # 0.0 ~ 1.0，回答置信度
    
    # === 工具调用记录 ===
    tool_calls: list[dict]             # 本次调用过的工具列表
    """
    [
        {
            "tool_name": "get_leave_balance",
            "arguments": {"user_id": "EMP001"},
            "result": {"annual_leave_remaining": 7},
            "success": True
        }
    ]
    """
    
    # === 知识来源 ===
    retrieved_chunks: list[dict]       # RAG 检索到的文档片段（用于溯源）
    """
    [
        {
            "source": "hr_policy_v2.pdf",
            "chunk_index": 3,
            "content": "...",           # 检索到的片段
            "relevance_score": 0.92
        }
    ]
    """
    
    # === 交割信息 ===
    handoff_suggestions: list[dict] | None  # 建议下一个处理的 Agent
    """
    当 Agent 判断问题涉及其他领域时，返回建议：
    [
        {
            "target_agent": "it_agent",
            "reason": "用户同时提到了设备报修需求",
            "context_summary": "已处理请假审批，待处理设备报修",
            "unresolved_items": ["创建 IT 工单"]
        }
    ]
    """
    
    # === 状态 ===
    status: str = "success"            # "success" | "partial" | "failed"
    error: str | None = None           # 错误信息（status 不为 success 时填充）
    
    # === 元信息 ===
    processing_time_ms: int = 0        # 处理耗时
    tokens_used: int = 0               # token 消耗
```

## 3. Agent 注册与路由

### 3.1 注册表

每个 Specialist Agent 在系统启动时向 Orchestrator 注册：

```python
@dataclass
class AgentRegistration:
    """Agent 注册信息"""
    agent_name: str                    # 唯一标识（hr_agent / it_agent / legal_agent）
    display_name: str                  # 展示名称（HR 专家 / IT 专家 / 法务专家）
    description: str                   # 能力描述（Orchestrator 用此做意图路由）
    """
    示例：
    "负责回答人事制度、考勤、请假、薪酬福利相关问题，
     可查询假期余额、提交请假审批。不处理技术和设备问题。"
    """
    tool_descriptions: list[str]       # 工具描述列表（用于 Agent 理解自身能力）
    """
    [
        "get_leave_balance(user_id) - 查询员工年假/病假剩余天数",
        "submit_leave_request(user_id, type, dates) - 提交请假申请"
    ]
    """
    input_schema: dict = None          # 可选：如果该 Agent 有特殊输入要求
    output_schema: dict = None         # 可选：如果该 Agent 有特殊输出要求
```

**注册表示例（硬编码在 Orchestrator 中）：**

```python
AGENT_REGISTRY = {
    "hr_agent": AgentRegistration(
        agent_name="hr_agent",
        display_name="HR 专家",
        description="负责回答人事制度、考勤、请假、薪酬福利相关问题...",
        tool_descriptions=[
            "get_leave_balance(user_id) - 查询员工年假/病假剩余天数",
            "submit_leave_request(user_id, leave_type, start_date, end_date, reason) - 提交请假申请"
        ]
    ),
    "it_agent": AgentRegistration(
        agent_name="it_agent",
        display_name="IT 专家",
        description="负责回答 IT 设备、软件安装、报修、网络问题...",
        tool_descriptions=[
            "check_ticket_status(ticket_id) - 查询工单状态",
            "create_ticket(user_id, issue_type, description, priority) - 创建新的 IT 工单",
            "check_device_inventory(device_type) - 查询设备库存"
        ]
    ),
    "legal_agent": AgentRegistration(
        agent_name="legal_agent",
        display_name="法务专家",
        description="负责回答合规、合同、数据保护、法律政策相关问题...",
        tool_descriptions=[
            "search_contract(keyword) - 搜索合同条款",
            "check_compliance(doc_summary) - 合规性初步检查"
        ]
    )
}
```

### 3.2 路由规则

Orchestrator 的意图路由实现方式（两种方案二选一）：

| 方案 | 做法 | 选型建议 |
|------|------|---------|
| **LLM Routing** | 把 `AgentRegistration.description` 列表塞进 prompt，让 LLM 选 | 灵活、冷启动快，初期推荐 |
| **Classifier Routing** | 训练/配置一个轻量分类器做意图映射 | 更稳定但需要标注数据，后期可升级 |

**LLM Routing Prompt 示意：**

```
你是一个问题路由专家。根据用户的问题，从以下 Agent 中选择最合适的来处理：

{agent_registrations}

用户问题：{query}

如果问题涉及多个领域，按重要性排序返回最多 2 个 Agent。

请返回 JSON：{"primary": "agent_name", "secondary": "agent_name | null"}
```

## 4. 工具协议

### 4.1 工具定义

每个工具使用统一格式定义：

```python
@dataclass
class ToolDef:
    """工具定义"""
    name: str                                 # 工具唯一名称
    description: str                          # 工具描述（给 LLM 看）
    parameters: dict                          # JSON Schema 格式的参数定义
    implementation: callable                  # 实际执行函数（mock 或真实）
```

**示例：**

```python
GET_LEAVE_BALANCE = ToolDef(
    name="get_leave_balance",
    description="查询员工年假/病假剩余天数。输入员工ID，返回假期余额。",
    parameters={
        "type": "object",
        "properties": {
            "user_id": {
                "type": "string",
                "description": "员工 ID，如 EMP001"
            }
        },
        "required": ["user_id"]
    },
    implementation=get_leave_balance_mock  # mock 函数
)
```

### 4.2 工具调用格式

Agent 内部使用 ReAct 模式调用工具：

```
# Agent 内部循环（不暴露给外部）
Thought: 用户想查年假余额，我需要调用 get_leave_balance 工具
Action: get_leave_balance({"user_id": "EMP001"})
Observation: {"annual_leave_remaining": 7, "sick_leave_remaining": 5}
Thought: 我现在有了假期数据，可以回答用户了
Final Answer: 你目前年假还剩 7 天...
```

### 4.3 Mock 工具规范

所有 MVP 工具使用 mock 实现，规则：

```python
def get_leave_balance_mock(user_id: str) -> dict:
    """Mock 工具：返回模拟假期数据"""
    MOCK_DATA = {
        "EMP001": {"annual_leave_remaining": 7, "sick_leave_remaining": 5},
        "EMP002": {"annual_leave_remaining": 12, "sick_leave_remaining": 3},
        "DEFAULT": {"annual_leave_remaining": 5, "sick_leave_remaining": 5},
    }
    return MOCK_DATA.get(user_id, MOCK_DATA["DEFAULT"])
```

- 每个工具维护 3-5 组预定义 mock 数据
- 未知输入返回合理默认值，不抛异常
- mock 数据在 `tools/mock_data.py` 中集中管理

## 5. 跨 Agent 交割协议

### 5.1 触发条件

Orchestrator 在以下情况触发跨 Agent 交割：

1. **AgentResponse.handoff_suggestions 非空** — Specialist Agent 主动建议转交
2. **Orchestrator 判断用户问题涉及多领域** — 初次路由时就识别出跨域需求

### 5.2 交割流程

```
Orchestrator 识别跨域问题
       │
       ├── 方案 A：串行编排（默认）
       │    Agent A 先处理 → 结果传给 Agent B → 汇总
       │
       ├── 方案 B：并行编排（P2 功能）
       │    Agent A 和 Agent B 同时处理 → Orchestrator 汇总
       │
       └── 方案 C：主次编排
            Agent A 为主，遇到需 Agent B 的子问题才调用
```

### 5.3 串行交割上下文传递

**交割上下文格式**（传递到下一个 Agent 的 `handoff_context`）：

```python
{
    "from_agent": "hr_agent",                     # 上一个 Agent
    "resolved_items": [                           # 已处理完成的部分
        "病假已审批，按 80% 工资计算"
    ],
    "unresolved_items": [                         # 待处理的部分
        "笔记本报修需要 IT 处理"
    ],
    "pending_tasks": [                            # 待执行的操作
        {"tool": "create_ticket", "params": {"user_id": "EMP001", "issue_type": "硬件"}}
    ],
    "shared_context": {                           # 共享信息
        "user_id": "EMP001",
        "department": "Engineering"
    }
}
```

### 5.4 汇总格式

Orchestrator 汇总多个 Agent 的结果：

```python
{
    "final_answer": "已完成以下处理：\n\n"
                    "**HR 处理：** 病假已审批，工资按 80% 计算。\n\n"
                    "**IT 处理：** 报修工单已创建，工单号 #1024，预计明天上门。",
    "sources": [
        {"agent": "hr_agent", "chunks": [...]},
        {"agent": "it_agent", "chunks": [...]}
    ],
    "tool_calls": [
        {"agent": "hr_agent", "tool": "submit_leave_request", ...},
        {"agent": "it_agent", "tool": "create_ticket", ...}
    ]
}
```

## 6. 错误处理协议

### 6.1 Agent 级别

| 错误场景 | Agent 行为 | 返回的 status |
|---------|-----------|-------------|
| RAG 检索无结果 | 尝试工具调用兜底，仍然无结果则返回"未找到相关信息" | `partial` |
| 工具调用失败 | 重试 1 次，仍失败则跳过该工具，继续处理 | `partial` |
| LLM 调用超时 | 返回已收集的信息 + 错误标记 | `partial` |
| 完全无法处理 | 返回错误信息 | `failed` |

### 6.2 Orchestrator 级别

| 场景 | Orchestrator 行为 |
|------|------------------|
| Agent 返回 failed | 重试 1 次，仍失败则走兜底：通用 LLM 回答 + 告知用户"部分功能不可用" |
| Agent 返回 partial | 使用已有信息回答，标注"部分信息可能不完整" |
| 跨 Agent 链路中断 | 已完成的部分正常返回，中断部分提示用户 |
| 意图识别置信度低 | 追问用户澄清，而不是硬分给某个 Agent |

### 6.3 重试策略

```python
RETRY_CONFIG = {
    "max_retries": 1,                # 最大重试次数
    "retry_delay_ms": 500,           # 重试间隔
    "timeout_per_agent_ms": 15000,   # 单个 Agent 超时
}
```

## 7. 对话记忆格式

### 7.1 存储格式

```python
@dataclass
class ConversationMemory:
    session_id: str
    messages: list[dict]             # [{"role": "user"/"assistant", "content": "..."}]
    metadata: dict                   # 本轮提取的实体、上下文等
    """
    {
        "user_id": "EMP001",
        "active_agents": ["hr_agent", "it_agent"],
        "resolved_items": [...],
        "pending_items": [...]
    }
    """
    max_turns: int = 10              # 最大保留轮数，超过则丢弃最早的
```

### 7.2 截断策略

- 保留最近 N 轮对话（默认 10 轮）
- 超过 N 轮时丢弃最早的，但保留其中提取的 `metadata`
- AgentRequest 中的 `conversation_history` 始终传完整保留轮数

## 8. 文件结构规范

每个 Agent 的文件结构必须遵循以下约定：

```
agents/
├── __init__.py
├── base_agent.py            # Agent 基类（定义 ReAct 循环）
├── orchestrator_agent.py    # Orchestrator
├── hr_agent.py              # HR Agent（继承 base_agent）
├── it_agent.py              # IT Agent
└── legal_agent.py           # 法务 Agent

tools/
├── __init__.py
├── base_tool.py             # ToolDef 定义
├── mock_data.py             # 所有 mock 数据集中管理
├── hr_tools.py              # HR 工具
├── it_tools.py              # IT 工具
└── legal_tools.py           # 法务工具

rag/
├── __init__.py
├── loader.py                # 文档加载
├── chunker.py               # 分块
├── embedder.py              # 向量化
├── retriever.py             # 检索（混合）
├── reranker.py              # 重排序
└── knowledge_base.py        # 知识库管理（每个 Agent 一个实例）

protocol/
├── __init__.py
├── types.py                 # AgentRequest / AgentResponse / AgentRegistration 定义
├── handoff.py               # 交割上下文构造器
└── errors.py                # 错误类型定义
```

## 9. 验收标准

每个 Specialist Agent 必须通过以下测试才算对接完成：

| 测试 | 描述 | 预期 |
|------|------|------|
| T1: 标准请求 | 收到 AgentRequest，返回 AgentResponse | status = "success"，answer 非空 |
| T2: 工具调用 | Agent 在需要时调用至少一个 mock 工具 | tool_calls 非空，success = True |
| T3: RAG 检索 | Agent 能基于知识库回答问题 | retrieved_chunks 非空 |
| T4: 错误兜底 | 检索无结果时给出合理回复 | status 为 partial，不抛异常 |
| T5: 跨 Agent 提议 | 遇到跨域问题时返回 handoff_suggestions | handoff_suggestions 格式正确 |
| T6: 交割上下文 | 收到 handoff_context 正确处理 | resolved_items 被理解，继续处理 unresolved |
