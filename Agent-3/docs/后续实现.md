# 企业级多 Agent 系统后续实现与细节优化文档

## 1. 文档目的

本文档用于覆盖和沉淀本项目后续实现方向，重点不再停留在“功能是否能跑”，而是逐步把系统从 Demo 形态优化为更完整、更稳定、更可解释、更接近真实企业场景的 Agent 工作台。

后续优化的核心目标是：

- 让 Agent 不只是回答问题，而是能稳定执行企业流程。
- 让前端不只是聊天窗口，而是企业工作台。
- 让 RAG 不只是检索文档，而是能给出可信、可追溯、可评估的答案。
- 让工具调用、审批、通知、Trace、反馈形成闭环。
- 让系统具备可演示、可测试、可扩展、可继续工程化的基础。

---

## 2. 当前系统定位

当前项目是一个企业级多专家 Agent 系统，主要面向 HR、IT、法务、财务等企业内部服务场景。

系统不应被理解为单纯的 ChatBot，而应被理解为：

```text
统一企业入口
→ 意图识别
→ 多 Agent 路由
→ RAG 检索
→ 工具调用
→ 业务状态变更
→ 审批/通知/追踪
→ 结果反馈
```

因此，后续实现应围绕“企业任务执行闭环”展开。

# 阶段一：稳定可用

## 5. 启动、配置与环境一致性

### 5.1 问题

当前系统依赖较多，但本地环境可能出现：

- `uvicorn` 未安装
- `.env` 缺失
- `TRACE_ENABLED` 未开启
- 前端从错误地址打开导致 API 不通
- Python 命令在 Windows 环境不一致

### 5.2 后续实现

需要补齐统一启动方式：

1. 明确推荐启动命令。
2. 明确依赖安装方式。
3. 明确 `.env` 示例。
4. 明确前端必须通过 FastAPI 静态服务访问。
5. 增加启动前自检脚本。

### 5.3 建议实现

新增或完善脚本：

```text
scripts/dev_check.py
scripts/start_api.py
scripts/init_db.py
```

启动前检查：

- Python 版本
- 依赖是否安装
- `.env` 是否存在
- `LLM_API_KEY` 是否配置
- `data/enterprise.db` 是否存在
- `traces/` 是否可写
- 前端目录是否存在

### 5.4 验收标准

- 新用户按 README 可以一次启动成功。
- 缺少依赖时提示明确。
- 缺少 API key 时提示明确。
- 打开首页后不会误显示“后端离线”。

## 6. 执行追踪 Trace 完整化

### 6.1 当前问题

Trace 容易出现以下问题：

- 没开启 `TRACE_ENABLED` 时不会写入。
- 写入 `.jsonl`，但查询接口如果只读 `.json` 会查不到。
- SSE 事件类型扩展后，Trace model 如果限制过死会记录失败。
- 前端 Trace 面板目前展示偏原始，需要结构化展示。

### 6.2 后续实现

Trace 应成为系统的核心可观测能力，而不是附属功能。

每次对话至少记录：

```text
trace_id
session_id
query
start_time
end_time
status
total_tokens
events[]
```

事件至少包括：

```text
routing
retrieval
confirm_tool
tool_call
answer_chunk
answer
error
```

### 6.3 前端优化

执行追踪面板应展示为时间线，而不是大段 JSON。

建议结构：

```text
Trace 卡片
├─ 用户问题
├─ 状态 / 耗时 / tokens
├─ 路由目标
├─ 事件时间线
│  ├─ 路由
│  ├─ 检索
│  ├─ 工具确认
│  ├─ 工具执行
│  ├─ 回答生成
│  └─ 完成
└─ 展开查看原始 JSON
```

### 6.4 后端优化

建议增加：

```text
GET /api/traces
GET /api/traces/{trace_id}
DELETE /api/traces
```

并支持查询参数：

```text
?status=success
?session_id=xxx
?limit=50
```

### 6.5 验收标准

- 每次对话后都能看到一条 Trace。
- Trace 面板无需看 JSON 也能理解执行过程。
- 错误请求能记录 error trace。
- SSE 和 POST 两种对话方式都能记录 Trace。

## 9. Human-in-the-loop 工具确认

### 9.1 当前问题

系统已有 `confirm_tool` 事件，但真实审批式确认还不完整。

对于会改变数据库状态的操作，不应全部由 LLM 自动执行。

### 9.2 需要确认的操作

必须确认：

- 提交请假
- 删除申请
- 修改申请
- 审批通过
- 审批驳回
- 提交报销
- 创建高优先级 IT 工单

可以直接执行：

- 查询年假余额
- 查询制度
- 查询申请状态
- 查询通知

### 9.3 后续实现方案

工具定义中增加风险等级：

```python
risk_level: "read" | "write" | "approval" | "admin"
requires_confirmation: bool
```

执行流程：

```text
LLM 请求调用工具
→ Tool Agent 判断是否需要确认
→ 前端展示确认卡片
→ 用户确认
→ 后端执行工具
→ 返回结果
```

### 9.4 前端确认卡片

确认卡片应展示：

```text
工具名称
操作说明
参数
影响范围
确认 / 取消
```

### 9.5 验收标准

- 读操作无需确认。
- 写操作必须确认。
- 用户取消后不会修改数据库。
- Trace 中记录确认与取消行为。

# 阶段三：能力增强

## 11. RAG 质量优化

### 11.1 当前问题

RAG 虽然可用，但还需要提高准确性、可解释性和可评估性。

可能问题包括：

- 检索结果不稳定
- chunk 粒度不一定合理
- 多问题合并时容易只回答一部分
- 引用来源展示不够清晰
- 无检索结果时回答策略不统一

### 11.2 后续优化方向

#### 11.2.1 Query Rewrite

对复杂问题进行拆解，例如：

```text
我的年假余额和报销标准分别是多少？
```

应拆为：

```text
1. 查询我的年假余额
2. 查询报销标准
```

#### 11.2.2 Hybrid Retrieval

结合：

- 向量检索
- BM25
- 关键词规则
- RRF 融合

#### 11.2.3 Rerank

对召回结果进行重排序，提高 top-k 准确性。

#### 11.2.4 Source Citation

回答中应展示来源：

```text
根据《年假管理制度》第 X 条...
根据《报销管理制度》...
```

#### 11.2.5 No-answer 策略

当知识库无依据时，应明确说明：

```text
当前知识库未找到相关依据，建议联系对应部门。
```

而不是编造答案。

### 11.3 验收标准

- 复杂问题能拆分回答。
- 回答能看到来源。
- 无依据时不胡编。
- RAG 命中结果能在 Trace 和前端卡片中查看。

## 12. 多 Agent 协同优化

### 12.1 当前问题

当用户提出跨领域问题时，例如：

```text
我的年假余额和报销标准分别是多少？
```

### 12.2 后续实现

```text

multi_agent_parallel
planner_dag
handoff
```

### 12.3 推荐策略

简单单领域问题：

```text
单 Agent
```

跨领域并列问题：

```text
多 Agent 并行
```

依赖型多步骤任务：

```text
Planner DAG
```

Agent 判断自己无法完成：

```text
Handoff
```

### 12.4 汇总回答

多 Agent 结果需要统一汇总：

```text
HR 部分：...
财务部分：...
下一步建议：...
```

### 12.5 验收标准

- 跨领域问题不会只回答一半。
- 前端能显示多个 Agent 的参与情况。
- Trace 能看到并行 Agent 执行链路。

## 13. Planner DAG 重新启用方案

### 13.1 当前状态

Planner 默认关闭，原因是之前复杂问题走 `planner_dag` 后前端显示不友好，出现置信度为 0 的问题。

### 13.2 后续方向

Planner 不应废弃，而是需要更规范地接入。

### 13.3 Planner 输出结构

Planner 应输出稳定结构：

```json
{
  "task_type": "multi_domain_query",
  "steps": [
    {"agent": "hr_agent", "query": "查询年假余额"},
    {"agent": "finance_agent", "query": "查询报销标准"}
  ],
  "execution_mode": "parallel",
  "confidence": 0.86
}
```

### 13.4 前端展示

Planner 应显示为：

```text
任务拆解
├─ HR：查询年假余额
└─ 财务：查询报销标准
```

### 13.5 验收标准

- Planner 开启后不再出现 0% 置信度误导。
- Planner DAG 可视化清晰。
- Planner 适合复杂任务，不影响简单问答速度。

## 15. Review Agent 与答案质量控制

### 15.1 当前目标

Review Agent 用于降低幻觉、检查回答是否符合事实和工具结果。

### 15.2 后续检查项

Review Agent 应检查：

- 是否引用了检索材料
- 是否与工具结果矛盾
- 是否遗漏用户问题
- 是否误导用户执行不可用操作
- 是否存在越权建议

### 15.3 输出结构

```json
{
  "passed": true,
  "issues": [],
  "revised_answer": "...",
  "confidence": 0.91
}
```

### 15.4 验收标准

- 工具结果和最终回答一致。
- 多问题不会漏答。
- 无依据时会提醒，而不是编造。

Agent 系统不能只靠主观体验，需要评估：

- 路由是否正确
- 检索是否命中
- 回答是否准确
- 工具调用是否合理
- 是否漏答
- 是否幻觉

### 17.2 建议建立评估集

示例：

```json
{
  "query": "我今年年假还剩几天？员工ID: EMP001",
  "expected_agent": "hr_agent",
  "expected_tools": ["get_leave_balance"],
  "expected_contains": ["年假", "剩余"]
}
```

### 17.3 指标

建议跟踪：

```text
routing_accuracy
retrieval_hit_rate
tool_call_accuracy
answer_completeness
hallucination_rate
latency_ms
tokens_used
```

### 17.4 验收标准

- 每次核心改动后能跑评估。
- 路由准确率可量化。
- RAG 质量可量化。

## 20. 性能与稳定性

### 20.1 需要关注

- 首次启动索引加载耗时
- RAG 检索耗时
- LLM 响应耗时
- SSE 长连接稳定性
- 多 Agent 并行时资源消耗
- 数据库锁竞争

### 20.2 后续优化

建议加入：

- 检索缓存
- RAG 结果缓存
- LLM retry/backoff
- 工具调用超时
- SSE 心跳
- 后台任务队列
- 慢请求日志

### 20.3 验收标准

- 常规问题 3 秒内开始流式输出。
- 工具调用失败有明确错误。
- SSE 不会 2 秒误断。
- 后端异常不会导致前端白屏。

## 21. 安全与权限

### 21.1 重点风险

企业 Agent 系统尤其要注意：

- 越权查询
- 越权审批
- Prompt Injection
- 工具滥用
- 敏感数据泄露
- 前端 token 暴露
- 审计缺失

### 21.2 后续实现

需要加入：

- 工具权限声明
- 用户角色校验
- Prompt Injection 检测
- 敏感字段脱敏
- 审计日志
- 管理接口权限校验

### 21.3 验收标准

- 普通用户不能访问管理员接口。
- Agent 不能绕过权限调用工具。
- 外部文档中的恶意指令不能覆盖系统规则。
- 所有写操作可追溯。

# 25. 最终目标形态

最终系统应形成一个完整的企业 Agent 工作台：

```text
用户登录
→ 在对话中提出问题或任务
→ Orchestrator 判断任务类型
→ 单 Agent / 多 Agent / Planner DAG 执行
→ RAG 提供制度依据
→ Tool Agent 执行业务工具
→ 必要时用户确认
→ 数据库状态变更
→ 审批流推进
→ 通知发送
→ Trace 记录全过程
→ 用户反馈进入质量闭环
```

最终体验应达到：

- 用户能自然语言完成企业事务。
- 管理者能看到审批和执行过程。
- 开发者能通过 Trace 定位问题。
- 系统能通过反馈和评估持续优化。

这才是本项目区别于普通 RAG 问答 Demo 的核心价值。
