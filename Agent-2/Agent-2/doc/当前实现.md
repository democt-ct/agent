# 当前实现

本文档记录当前仓库里已经落地的实现，不展开产品设想，只描述现在实际在跑的流程、入口和行为。

## 1. 总体结构

当前实现是“前端展示 + FastAPI 后端 + 公共网页检索 + 高德 POI 检索 + 候选池 + 行程生成 + 前端直接写回行程”的组合。

- 前端主页面：`fastapi/static/index.html`
- 后端主流程：`fastapi/app.py`
- 候选池和路线生成数据最终都由后端生成并返回给前端展示
- 地图页面展示的是后端已经产出的 `itinerary`、`mapData`、`resolvedPlaces`
- 前端还会对回复中的地点做补充解析、补充写回和路线重算

## 2. 当前地点检索链路

现在的地点检索不是单一来源，而是混合式链路：

1. 用户输入进入后端
2. 后端判断是否需要轻量联网，必要时抓取网页上下文 `web_context`
3. `web_context` 会参与查询计划生成
4. 查询计划会同时驱动高德 POI 搜索
5. 网页中提取到的地点名会再次回流到 POI 搜索
6. 所有候选点进入去重、打分、筛选
7. planner 按天、按时段槽位生成行程
8. 前端根据返回的 itinerary 和 mapData 画路线与点位
9. 如果回复里还有未完全落图的泛地点需求，前端会补做 POI 解析，并在命中候选后直接写回正式行程

## 3. 网页上下文检索

### 3.1 触发方式

当前网页检索不再只依赖“搜索 / 攻略 / 小红书 / 微博”这类显式词。

后端会把以下情况也视为推荐型旅游请求，自动触发轻量联网：

- 有目的地
- 有兴趣点
- 有天数
- 有用户偏好
- 包含“推荐、路线、行程、怎么玩、怎么逛、适合、安排、规划、Day 1 / Day 2”等语义

相关实现主要在 `fastapi/app.py`：

- `has_web_context_intent(...)`
- `should_auto_web_search(...)`
- `build_web_search_queries(...)`
- `collect_web_context(...)`

### 3.2 网页检索输出

网页检索输出的是 `web_context`，包含：

- `queries`
- `items`
- 页面标题、摘要、URL 等内容

此外，当前实现还会从网页上下文里抽取可能的地点名，供后续 POI 回流。

## 4. 网页地点回流 POI

当前已经实现“网页提取地点名 -> 再次 POI 搜索”的补强链路。

具体做法是：

1. 从 `web_context` 的 title / snippet / text 中提取疑似地点名
2. 对抽出的地点名做去重
3. 用这些地点名再发起高德 POI 搜索
4. 将回流结果合并进候选池

相关实现主要在：

- `extract_place_name_candidates(...)`
- `extract_web_place_names(...)`
- `build_web_followup_poi_tasks(...)`
- `build_amap_candidate_pool(...)`

## 5. 查询计划

### 5.1 当前策略

查询计划现在是“基础类目 + 按偏好扩展”的模式，而不是平均铺开。

基础类目默认保留：

- `landmark`
- `food`

再根据用户兴趣、节奏、天数、距离偏好扩展：

- `nightview`
- `cafe`
- `park`
- `citywalk`
- `mall`
- `nature`
- `museum`

### 5.2 计划生成

`build_default_query_plan(...)` 会根据：

- `destination`
- `interests`
- `user_preferences`
- `trip_days`

生成查询类目顺序，然后只保留前几类最相关的类目。

`build_candidate_query_plan(...)` 会在默认规则基础上，结合 `web_context` 和网页抽取的地点名生成更贴近当前请求的查询计划。

### 5.3 查询任务

查询任务分两层：

- 普通 POI 任务：根据 query plan 生成
- 网页回流 POI 任务：根据网页抽取地点名额外补一轮

对应实现：

- `build_query_tasks_from_plan(...)`
- `build_web_followup_poi_tasks(...)`
- `build_amap_candidate_pool(...)`

## 6. 候选池

### 6.1 候选来源

当前候选池主要来源于：

- 高德 POI 搜索结果
- 网页回流 POI 搜索结果
- 未来也保留了网页搜索、静态池等接入位置

### 6.2 去重和打分

候选池当前不是简单“先到先留”，而是：

- 按 `sourceRef`、名称、分类、地理位置做去重
- 优先保留质量更高的候选
- 根据来源、坐标、地址、类别、置信度给候选排序

相关实现：

- `candidate_quality_score(...)`
- `dedupe_candidates(...)`

### 6.3 候选摘要

为了让复筛和 planner 更容易工作，候选池会先做一层摘要和均衡采样：

- 优先保留带地理位置的候选
- 保留更高置信度的点
- 控制单一类别过度集中

相关实现：

- `summarize_candidate_pool_for_llm(...)`

## 7. 复筛与偏好

当前还有一层候选复筛逻辑：

- 允许模型在候选池内给出偏好排序
- 不允许新增候选点
- 输出 `preferredCandidateIds`
- 输出 `avoidCandidateIds`
- 输出 `preferredCategories`

这样 planner 在选点时会更偏向更适合当前行程节奏的点，而不是机械地取高分前几名。

相关实现：

- `review_candidate_pool_selection(...)`
- `selectionHints`

## 8. Planner

### 8.1 现有生成方式

planner 当前已经不是“把候选按分数排序后直接取前几个”。

现在的生成方式是：

1. 先对候选做统一排序
2. 再按 day template 拆分
3. 每天按时段槽位挑选
4. 不同天使用不同模板
5. 结合近邻关系和类别重复惩罚控制行程质量

### 8.2 时段槽位

当前槽位为：

- `morning`
- `lunch`
- `afternoon`
- `evening`

每个槽位有对应的类别偏好，例如：

- morning 更偏 `landmark / museum / park / nature`
- lunch 更偏 `food`
- afternoon 更偏 `citywalk / cafe / mall / park`
- evening 更偏 `nightview / food / citywalk`

### 8.3 模板

当前模板包括：

- `arrival`
- `balanced`
- `foodie`
- `closing`
- `nature`

不同天会根据天数、兴趣、节奏选择不同模板。

### 8.4 选点规则

planner 选点时会综合：

- 候选置信度
- 是否有地理位置
- 是否命中偏好候选
- 是否命中避让列表
- 是否匹配当前时段槽位
- 是否与前一个点太远
- 是否重复使用过多同类点
- 是否符合 query plan 的类别优先级

相关实现：

- `select_planner_items(...)`
- `build_planner_output(...)`

### 8.5 地图数据

planner 会同时输出：

- `itinerary.days`
- `mapData.markers`
- `mapData.layers`
- `mapData.center`

地图上的点位用于前端切换“第几天”路线时显示。

## 9. 前端展示

当前前端主要在 `fastapi/static/index.html`。

已经落地的行为包括：

- 地点卡片支持按 Day 展示
- 地图支持按“第几天”切换路线
- 地图点位显示数字标记
- 地点卡片去掉了图片缩略图
- Day 1 / Day 2 界面里增加了流程指示，显示距离信息
- 模型输出里的 `**` 会被清洗掉后再显示
- 行程卡片在地点名下方默认只展示地址，不再展示较长的安排说明
- 地图侧的泛地点需求不再先停在“待选择候选”，而是先查 `/api/planner/place-candidates`，再把首个候选直接写回行程
- 备选地点仍然保留独立面板，但加入某一天时走的是 `direct_add` 写回，而不是重新发起一整轮大模型重规划
- 拖拽重排只对“当前选中那一天、已确认、带 `poi_id` 且可解析坐标”的正式行程点开放

## 10. 当前前端补充解析与额度控制

当前前端有一条补充解析链路，用来把回复中的地点名和泛地点需求转成可落图 POI：

- 明确地点走 `/api/map/poi/resolve`
- 泛地点走 `/api/planner/place-candidates`
- 行程写回后会触发地图刷新与路径重算

为了减少高德额度消耗，当前版本已经增加两层前端缓存：

- `resolvePlaceCandidate(...)` 的解析结果缓存
- `/api/planner/place-candidates` 的候选结果缓存

这意味着：

- 重复打开同一个会话
- 同一条回复反复同步地图
- 同一组泛地点需求重复补点

这些场景下，不会再无条件重复请求高德。但首次同步一条新回复时，仍可能发生多次位置解析和路线计算。

## 11. 当前行程写回与校验

当前前后端已经形成一套可直接编辑的写回链路：

- 搜索结果可直接加入某一天
- 备选点可直接加入某一天
- 泛地点需求会优先自动补成正式 POI 并直接写回
- 写回后会更新 `manual_order_poi_ids`、地图点位和路径

后端校验阶段还会额外处理两类约束：

- 住宿类 POI 不应混入普通游玩路线
- 第 `1` 天与第 `2` 天的正式行程点不应出现同地址重复；如果重复，保留前一天并剔除第二天重复点

## 12. 当前入口

当前最关键的主入口是：

- `generate_itinerary(...)`
- `build_candidate_query_plan(...)`
- `build_amap_candidate_pool(...)`
- `build_planner_output(...)`

用户发起规划请求后，后端会把网页上下文、查询计划、候选池、复筛结果和模板化 planner 串成一个完整的行程输出。

## 13. 当前状态总结

现在这套实现已经具备：

- 轻量联网补充
- 网页地点提取回流 POI
- 基础类目 + 偏好扩展的查询计划
- 候选池去重与排序
- 类别平衡的复筛
- 按模板和时段槽位生成行程
- 地图按天切换展示
- 泛地点自动补点并直接写回
- 第 1 / 第 2 天重复地址修复
- 前端位置解析缓存与候选缓存

如果后续继续优化，最值得继续补强的方向是：

- 网页内容向 POI 候选的转化再更强一些
- planner 的近邻连贯性和日内节奏再细一点
- TS 侧实现与 FastAPI 主链路继续对齐
