
请修复当前旅游 Agent 中“搜索词/类别词被当作真实路线地点”的问题。注意不要针对任何城市写死规则，也不要写固定城市路线。请做通用机制。

核心问题：
当前系统可能会把搜索 query、类别词或泛地点名直接放进最终路线，例如“步行街”“美食”“景点”“住宿”“酒店”等。这些词可以作为搜索关键词或分类标签，但不能作为 route item.name 出现在用户路线中。

请修改以下逻辑：

1. 区分 query、category、real POI

- search keyword 只能作为搜索参数，不能进入 candidate pool。
- category label 只能作为分类，不能进入 route items。
- route item 必须是真实地点，且至少满足：
  - 有地图 POI id；或
  - 有经纬度和 map/amap 来源；或
  - 来自 curated/web_verified 且是明确地点名。
- food_signature 只能作为“吃什么”的说明，不能直接作为路线地点。
- hotel/lodging 只能作为住宿信息或起终点，不能混入 playable items。

2. 增加通用名称过滤
   请实现：

- is_generic_query_term(name)
- is_valid_route_item_name(name, poi=None)
- validate_route_item(item)

需要过滤：

- 泛类别词；
- 搜索关键词；
- 攻略类长句；
- 没有验证来源的候选；
- 内部状态词；
- 只有 category 没有真实 POI 的 item。

3. 修复候选池构建

- 泛搜索结果只能进入 discovery_candidates。
- discovery_candidates 不能直接进入 final candidate pool。
- 必须经过真实地点名抽取、名称校验、地图验证、去重、POI 类型过滤后，才能进入候选池。
- 禁止把搜索 query 本身作为候选 POI。

4. 修复 slot fallback

- fill_day_slot 不能返回 category name 或 query term。
- fallback 必须返回具体 POI。
- 如果只有类别需求，应再次搜索并验证具体 POI。
- 如果找不到具体 POI，可以跳过该 slot 并给 warning，但不能用泛词硬填。

5. 修复餐饮时间段
   请实现：
   is_food_suitable_for_meal_slot(poi, meal_slot)

要求：

- breakfast 不能推荐火锅、串串、烧烤、酒吧、夜宵类。
- lunch/dinner 才允许重餐饮。
- 不合适的餐饮不能进入对应 meal slot。

6. 修复住宿展示

- hotel/lodging 单独进入 lodging 字段。
- hotel 可以作为起点/终点。
- hotel 不计入 playableItemCount。
- hotel 不参与 sightseeing cluster。
- hotel 不作为普通游玩地点展示。

7. 修复 day theme

- day theme 不能出现 closing、fallback、unknown、default、cluster 等内部状态词。
- 如果无法生成主题，使用通用安全主题，例如“轻松城市游”。
- theme 应根据当天真实内容生成。

8. 增强 validator
   validator 必须检查：

- route item 是否为泛词或搜索词；
- item 是否缺少 poi_id/经纬度/可信来源；
- hotel 是否混入 playable items；
- breakfast 是否选择不合理餐饮；
- day theme 是否包含内部状态词；
- slot fallback 是否使用了 category/query 作为地点。

如果发现问题，必须返回 is_valid=false，并给出 repair_suggestions。
