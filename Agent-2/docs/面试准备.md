# Travel Agent Worker 面试准备

本文档覆盖该项目面试可能被问到的核心知识点。建议结合源码交叉阅读。

---

## 1. 项目一句话概述

一个部署在 Cloudflare Workers 上的 **AI 旅行规划 Agent**，用户用自然语言描述旅行需求，系统调用 LLM 理解意图、构建候选池、生成日程计划，并通过高德地图 API 完成 POI 落点和路线规划，最终返回可交互的地图行程。

---

## 2. 系统架构

```
用户浏览器
  │
  ├─ GET / → 静态资源 (index.html + JS + CSS)
  ├─ POST /chat → 对话入口（收集需求 / 生成行程 / 重规划）
  ├─ POST /map/search → 高德 POI 搜索
  ├─ POST /api/map/route/plan → 高德路径规划
  └─ GET /sessions → 会话管理
  │
  ▼
Cloudflare Worker (src/index.ts)  ← 单一入口，路由分发
  │
  ├─ LLM 调用 (ModelScope / OpenAI 兼容 API)
  │    ├─ 需求解析 (requirementInterpreter)
  │    ├─ 候选池构建 (candidatePoolBuilder)
  │    ├─ 城市签名种子 (citySignatureSeeder)
  │    └─ 候选审查 (candidatePlanner)
  │
  ├─ 高德 API (McpClient)
  │    ├─ POI 搜索 (text search)
  │    ├─ 路线规划 (walking / driving)
  │    ├─ 逆地理编码 (坐标→城市)
  │    └─ 天气查询
  │
  ├─ 规划引擎 (travelPlanningPipeline)
  │    ├─ 候选池构建
  │    ├─ 城市签名池匹配
  │    ├─ 动态行程编排 (dynamicItineraryPlanner)
  │    └─ 路线校验 (routeValidator)
  │
  └─ D1 数据库 (SQLite)
       ├─ sessions
       ├─ trip_requirements
       ├─ itinerary_drafts
       ├─ conversation_messages
       ├─ user_preferences
       └─ user_place_notes
```

### 为什么选择 Cloudflare Workers？

- **全球边缘部署**：Worker 运行在 Cloudflare 全球网络，用户就近访问
- **零运维**：不需要管理服务器、容器、K8s
- **D1 集成**：内置 SQLite 兼容数据库，省去单独部署数据库
- **静态资源同域托管**：`[assets]` 绑定使前端和 API 同域，天然避免跨域问题
- **免费额度充足**：每天 10 万请求，小项目零成本

## 3. Cloudflare Workers 核心知识

### 3.1 Worker 运行时特点

- 基于 V8 引擎的 Service Worker API，不是 Node.js
- 每个请求在独立 isolate 中执行，无共享状态
- Web API 子集可用：`fetch`、`Request`、`Response`、`URL`、`crypto`
- 不可用：`fs`、`net`、`child_process` 等 Node.js API
- CPU 时间限制（免费 10ms/请求，付费 50ms，但 I/O 不计入）
- 请求 wall time 上限约 30 秒

### 3.2 绑定 (Bindings)

```toml
# wrangler.toml
[[d1_databases]]
binding = "DB"              # D1 数据库绑定

[assets]
binding = "ASSETS"          # 静态资源绑定
```

绑定通过 `env` 参数注入到 `fetch` 函数：

```typescript
export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    const db = env.DB;           // D1Database
    const assets = env.ASSETS;   // Fetcher (静态资源)
  }
};
```

### 3.3 环境变量

- `wrangler.toml` 的 `[vars]` → 部署时写入 Cloudflare
- `.dev.vars` → 仅本地 `wrangler dev` 时生效，不部署
- 敏感信息（API Key）放 `[vars]` 而非 `wrangler.toml`，或使用 Cloudflare Secrets

### 3.4 wrangler CLI

```bash
npx wrangler dev          # 本地调试 (默认 :8787)
npx wrangler deploy        # 部署到 Cloudflare
npx wrangler tail          # 查看生产日志
npx wrangler d1 execute DB --file=migrations/0001_init.sql  # 执行迁移
```

## 4. D1 数据库 (SQLite)

### 4.1 特点

- Cloudflare 的 Serverless SQLite 服务
- 兼容 SQLite 语法和大部分标准 SQL
- 自动备份、HTTP API 访问
- 适合读多写少的场景
- 单个数据库有存储限制，不支持大事务

### 4.2 本项目表设计

| 表 | 用途 | 关键字段 |
|---|---|---|
| `sessions` | 会话管理 | id, visitor_id, current_requirement_version, current_itinerary_version |
| `trip_requirements` | 需求快照（不可变） | session_id, version, raw_input, structured_payload_json |
| `itinerary_drafts` | 行程快照（不可变） | session_id, version, itinerary_json, generator_type |
| `conversation_messages` | 对话记录 | session_id, role, content, metadata_json |
| `user_preferences` | 用户偏好 | owner_type, owner_id, preferred_pace, interests_json |
| `user_place_notes` | 用户标注地点 | session_id, query, poi_json, rating |

### 4.3 版本控制策略

需求和行程都采用 **不可变版本记录**（append-only），不更新旧记录：

```
每次用户修改需求 → INSERT 新 version → bump current_requirement_version
每次生成/重规划行程 → INSERT 新 version → bump current_itinerary_version
```

好处：可追溯历史、不丢数据、支持回滚对比。

### 4.4 面试常见问题

**Q: 为什么不用 MySQL/PostgreSQL？**
A: Workers 不支持 TCP 直连传统数据库。D1 是 Cloudflare 原生的 Serverless SQLite，零延迟、零运维、与 Worker 同生命周期。

**Q: SQLite 的并发限制怎么处理？**
A: D1 内部做了读写分离和 WAL 优化。本项目请求量不大，读多写少，完全够用。高并发场景可以考虑用 Cloudflare Hyperdrive + 外部 PG。

```typescript
// 3. 代码默认值

export function resolveLlmRuntimeConfig(env: Env): LlmRuntimeConfig {
  // provider → baseUrl → model → apiKey 的 fallback 链
}
```

### 5.2 双端降级策略 (`openaiResponsesClient.ts`)

```
非 api.openai.com:
  → 直接调 /chat/completions（OpenAI 兼容 API）

api.openai.com (原生 OpenAI):
  → 先试 /responses (支持 strict JSON Schema)
  → 失败则降级 /chat/completions
```

### 5.3 Structured Output 方案

由于 ModelScope 等第三方 API 不支持 `response_format: { type: "json_object" }`，采用 **Prompt 内嵌 Schema** 的方式：

```typescript
const prompt = [
  userInput,
  "",
  "Return valid JSON only.",
  `Schema name: ${schemaName}`,
  `Schema: ${JSON.stringify(schema)}`
].join("\n");
```

### 5.4 需求解析 (`requirementInterpreter.ts`)

双重策略：

1. **规则解析** (`requirementParser.ts`)：正则提取天数、预算、兴趣、约束等
2. **LLM 解析**：调用模型提取结构化需求，失败时自动 fallback 到规则解析

```typescript
// LLM 成功 → 使用 LLM 结果
// LLM 失败 → console.warn → 回退规则解析
try {
  const llmPayload = await client.createStructuredJson({...});
  // ...
} catch (error) {
  console.warn("falling back to rule parser", error);
  // 返回规则解析结果
}
```

### 5.5 LLM 调用的 4 个场景

| 调用方 | 用途 | 输出 |
|---|---|---|
| `requirementInterpreter` | 自然语言→结构化需求 | destination, trip_days, interests, budget |
| `citySignatureSeeder` | 生成城市必游清单 | must_visit_attractions, famous_foods, night_options |
| `candidatePlanner` | 生成搜索关键词计划 | 每个 POI 分类的搜索词 |
| `candidatePlanner` | 审查候选池质量 | 推荐/排除的候选 ID |

**面试要点**：每个 LLM 调用都有 fallback 机制，不会因为模型失败导致整个链路中断。

## 6. 高德地图 API 集成 (`McpClient`)

### 6.1 调用的 API

| API | 用途 | 关键参数 |
|---|---|---|
| `v3/place/text` | POI 关键词搜索 | keywords, city, offset |
| `v3/direction/walking` | 步行路径规划 | origin, destination |
| `v3/direction/driving` | 驾车路径规划 | origin, destination |
| `v3/geocode/regeo` | 逆地理编码（坐标→城市） | location |
| `v3/weather/weatherInfo` | 天气查询 | city, extensions |

### 6.2 Key 的三种类型

```
AMAP_BROWSER_KEY    → 前端 JS SDK 用（地图渲染 + 浏览器端搜索）
AMAP_SECURITY_JS_CODE → 前端 JS SDK 安全密钥
AMAP_WEB_SERVICE_KEY  → 后端 API 调用用（POI 搜索、路径规划）
```

**注意**：浏览器 Key 不能用于服务端 API 调用，否则会忽略 `city` 参数。

### 6.3 关键词变体策略

为提高 POI 匹配率，对同一语义生成多个搜索变体：

```typescript
// 用户搜索 "成都 咖啡馆"
keywordVariants = [
  "成都 咖啡馆",
  "成都附近 咖啡馆",   // nearby scope
  "成都周边 咖啡馆",   // surrounding scope
  "成都近郊 咖啡馆"
];
// Promise.any() 并发请求，任意一个成功即返回
```

### 6.4 路线规划

- 按天按顺序连接 POI 点
- 两点之间调用高德步行/驾车路径
- 无 Key 时用 Haversine 公式计算直线距离作为降级
- 路线数据含：polyline（地图画线用）、距离（米）、耗时（秒）

## 7. 规划管线 (`travelPlanningPipeline`)

### 7.1 完整流程

```
用户输入 "想去成都玩3天，喜欢美食和自然风光"
  │
  ▼
① interpretRequirement  →  destination="成都", trip_days=3, interests=["美食","自然风光"]
  │
  ▼
② resolveSearchBasePayload → 处理定位、城市来源标注
  │
  ▼
③ findMissingFields → 检查 destination 和 trip_days
  │
  ▼
④ planTrip:
  ├─ buildCitySignatureSeed (LLM) → 成都必游景点、特色美食、夜间选择
  ├─ buildCandidateQueryPlan (LLM) → 每个分类的搜索关键词
  ├─ buildCandidatePool:
  │   ├─ web search (DuckDuckGo) → 内容信号
  │   ├─ AMap POI search → 候选 POI
  │   └─ citySignaturePool → 城市签名校验
  ├─ planDynamicItinerary:
  │   ├─ anchorClusterPlanner → 锚点聚类编排
  │   └─ 按天/时段分配 POI
  ├─ enrichPlannerOutputWithRoutes → 高德路径规划
  └─ routeValidator → 校验路线合理性
  │
  ▼
⑤ 返回: 天级行程 + 地图数据 (markers + polylines) + 候选池
```

### 7.2 候选池 (`candidatePoolBuilder`)

三个来源：

| 来源 | 方式 | 特点 |
|---|---|---|
| Content Search | DuckDuckGo 网页搜索 | 发现网红/攻略中的地点 |
| MCP POI Search | 高德 text search | 精确 POI + 坐标 |
| City Signature Pool | LLM 城市知识 + 静态库 | 必游景点/美食街/商圈 |

### 7.3 行程编排 (`dynamicItineraryPlanner`)

核心概念：
- **天 (day)**：第 1 天、第 2 天...
- **时段 (slot)**：上午、中午、下午、晚上
- **锚点 (anchor)**：每天的核心景点，其他 POI 围绕锚点布局
- **聚类 (cluster)**：美食聚类、夜生活聚类、购物聚类

编排策略：
- Light Comfort：轻松舒适，每天 2-3 个点
- Classic：经典必游，覆盖城市签名
- Food First：美食优先
- High Intensity：高强度打卡

### 7.4 面试要点

**Q: 为什么选择先搜再排的管线，而不是直接让 LLM 生成行程？**
A: LLM 的 POI 知识有时效性和幻觉问题（编造不存在的餐厅）。先用搜索 + 高德 API 获取真实 POI 数据，再用算法编排，保证行程中的每个地点都有真实坐标和地址。

**Q: 如果高德 API 没返回结果怎么办？**
A: 有多层降级：关键词变体重试 → web search 兜底 → 空候选池时触发 `planning_blocked`，告知用户而非返回假数据。

## 9. 常见面试追问

### Q: 为什么需求和行程用不可变版本记录？

避免更新丢失历史数据。用户可能想比较两版行程，或回滚到上一版。版本号让并发更新也能安全排序。

### Q: 如何处理 LLM 返回不合法 JSON？

1. `createStructuredJson` 的 Responses API 路径有 `strict: true`
2. Chat Completions fallback 用 `JSON.parse` 并 try/catch
3. `interpretRequirement` 外层有 fallback 到规则解析

### Q: 候选池为什么要做 content search（网页搜索）？

高德 API 擅长精确 POI 搜索，但不理解"网红打卡地""小众秘境"这类语义。DuckDuckGo 搜索能发现攻略文章中的地点名，补充高德搜不到的内容。

### Q: 如何处理高德 API 配额超限？

1. 前端缓存已解析的 POI，避免重复请求
2. 直线距离 Haversine 公式作为路径规划的降级
3. McpClient 所有方法返回 `warnings` 数组而非直接抛异常

### Q: 项目如果扩展到 10 万 DAU 会遇到什么问题？

- D1 并发限制：需要引入读写分离或缓存层
- Worker CPU 限制：规划管线需要拆分为多个 Worker 或使用 Queues
- LLM 调用延迟：需要缓存常见目的地的候选池
- 高德 API 配额：需要商业化 QPS 套餐 + 客户端缓存
