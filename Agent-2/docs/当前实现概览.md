# 当前实现概览

本文档记录截至当前版本，项目里已经落地可用的旅行规划、地图展示与行程编辑能力，重点描述“现在已经实现了什么”。

## 1. 总体能力

当前系统已经形成一条完整链路：

1. 用户通过聊天输入旅行需求。
2. 后端生成结构化行程 `itinerary`。
3. 前端从行程里提取地点，调用高德 POI 解析与路径规划接口。
4. 地图展示每天的地点、路线和下方地点列表。
5. 用户可以继续搜索地点、把备选点直接加入指定日期，并实时刷新当前路线。

## 2. 已实现的地图能力

### 2.1 行程自动落图

- 会从当前 `itinerary.days[].items[]` 中抽取地点候选。
- 对明确地点自动调用 `/api/map/poi/resolve` 获取坐标。
- 对泛化地点会调用 `/api/planner/place-candidates` 获取候选，并优先把首个候选直接写回正式行程。
- 已解析成功的地点会显示在地图和下方地点卡片中。
- 已识别但没有坐标的地点会显示为“待解析坐标”。
- 前端已增加地点解析缓存和泛地点候选缓存，重复打开同一会话或重复同步同一条回复时，不会无条件重复请求高德。

### 2.2 路线规划

- 路线不再从“我的位置”起步。
- 每天路线直接从当天第 `1` 个行程点开始。
- 路线点位和下方地点简介都按 `1, 2, 3...` 编号。
- 支持按“第几天”切换当天路线与地点列表。
- 2D / 3D 模式可切换。
- 3D 视角已收敛，不会再默认缩放过大。

### 2.3 地图空状态与定位

- 刚进入页面、没有路线时，地图默认尝试定位到用户当前位置。
- 如果浏览器定位失败，会尝试高德定位兜底。
- 如果仍然失败，则回退到默认城市中心点。
- 没有路线时不会自动展示默认搜索结果。
- 新建会话时会清空旧路线、旧搜索结果和旧搜索词，并重置地图状态。

## 3. 已实现的地点解析能力

### 3.1 城市清洗

- 已增加城市名清洗逻辑。
- 可以过滤 `第1`、`第2天` 这类误识别为城市的值。
- 可以清洗类似 `成都玩`、`上海旅游` 这类带尾缀的城市表达。

### 3.2 关键词兜底解析

- 地点解析会先尝试原始关键词。
- 对 `IFS美食`、`某某附近`、`某某周边` 这类语义表达，会自动生成更短的关键词变体继续解析。
- 如果最终是用锚点词落图，例如把 `IFS美食` 落到 `IFS` 附近，会在前端显示注释说明。

### 3.3 高德接口兼容

- 后端保留了高德 POI 查询主链路。
- 当主请求无结果时，会尝试兼容请求路径。
- 城市约束、标准化和排序逻辑已经统一接入。

## 4. 已实现的行程编辑能力

### 4.1 泛地点自动补点与直接写回

- 对“咖啡馆 / 商圈 / 美食街 / 夜景”这类泛地点需求，前端不再先停在“待选择候选”。
- 当前实现会先请求 `/api/planner/place-candidates`，取排序靠前的候选 POI，直接写回当前行程对应的天和时段。
- 写回后会重新同步地图和路线。

接口：

- `POST /api/planner/itinerary/place-select`

### 4.2 备选行程展示

- 助手回复里的“备选”段落会被抽取为备选地点列表。
- 备选区整体展示在地图下方。
- 每个备选点可选择“加入第1天 / 第2天 ...”。

### 4.3 备选加入当天

- 加入备选行程时，不再重新发给大模型做整轮重规划。
- 现在会直接把已解析好的 POI 写进指定那一天。
- 写入后只执行地图同步与路径重算，不再来回跑大模型。
- 这一步支持 `direct_add` 模式。

### 4.4 行程校验与写回约束

- 后端校验阶段会检查第 `1` 天和第 `2` 天是否出现同地址的正式行程点。
- 如果第 `2` 天地址与第 `1` 天重复，后端会保留前一天的点，并把第二天重复地址的点从正式行程中剔除，同时写入 `planner_warnings` 和 `validator_result`。
- 写回后的 `manual_order_poi_ids` 会同步刷新，保证地图路线和列表顺序一致。

相关实现：

- 前端加入逻辑：[fastapi/static/index.html](/D:/zhuomian/Agent-2/fastapi/static/index.html)
- 后端直接写入逻辑：[fastapi/core.py](/D:/zhuomian/Agent-2/fastapi/core.py)

## 5. 已实现的搜索能力

- 用户可以手动搜索任意地点。
- 搜索结果会单独显示为“搜索结果”。
- 手动搜索不会再清空当前行程路线。
- 行程地点、待解析点和手动搜索结果可以同时存在于页面中。

接口：

- `POST /map/search`
- `POST /api/map/poi/resolve`
- `POST /api/map/route/plan`

## 6. 已实现的会话能力

- 支持创建新会话。
- 支持恢复最近会话。
- 支持查看历史会话列表。
- 支持删除会话。
- 切换会话时会同步恢复对应聊天记录和最新行程。

接口：

- `POST /sessions`
- `GET /sessions`
- `GET /sessions/{session_id}/messages`
- `GET /sessions/{session_id}/itineraries/latest`
- `DELETE /sessions/{session_id}`

## 7. 当前前端表现

当前页面已经具备以下交互表现：

- 聊天区展示规划结果。
- 地图区展示路线、点位和状态提示。
- 地点列表支持按天查看。
- 待解析地点会明确提示。
- 泛地点会优先自动补成正式 POI，而不是先停在待选列表。
- 备选行程可直接挂入指定日期。
- 搜索结果与原路线共存显示。
- 行程卡片在地点名下方只保留地址，不再展示“第几天晚上”等说明文案。
- 地图列表里的拖拽排序只对“同一天内、已确认、带 `poi_id` 且有坐标”的正式行程点开放。

## 8. 当前后端能力边界

后端目前已经能支撑：

- 行程快照保存
- 行程地点解析
- 泛地点自动补点并直接写回
- 备选点直接加入某一天
- 第 `1` 天 / 第 `2` 天重复地址校验与修复
- 路径规划结果返回

但仍然依赖高德返回 POI 与定位结果，因此当高德配额、定位权限或网络状态异常时，前端仍可能进入降级提示状态。

## 9. 当前已知注意事项

- 地图能力依赖高德浏览器 SDK 与高德服务端接口。
- 用户定位依赖浏览器权限，若被拒绝会走回退逻辑。
- 某些语义地点虽然可以通过锚点兜底落图，但并不一定是精确商户位置，而是“附近落点”。
- 首次同步一条新回复时，前端仍可能触发多次位置解析与路线计算；当前版本已经通过前端缓存显著减少重复请求，但没有做到“全链路只请求一次”。
- 路由规划现在是基于当前行程内已落图点进行重算，不会自动重写整份文本行程描述。

## 10. 一句话总结

当前版本已经不只是“聊天生成文本行程”，而是具备了：

- 聊天生成行程
- 高德 POI 落图
- 按天路径规划
- 泛地点自动补点
- 备选点直接加入指定日期
- 搜索与原路线共存
- 会话级地图与行程状态管理

这意味着项目已经进入“可编辑、可落图、可重算”的交互阶段。
